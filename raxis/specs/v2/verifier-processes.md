> **Status.** Normative for V2.
>
> **Scope.** This spec is the canonical home for the mechanical witness flow:
> the `raxis-verifier` PID-1 binary, the verifier-VM lifecycle, the

## ¬ß1 ‚ÄĒ Why This Spec Exists

V2's Pure-Static Reviewer decision ([`planner-harness.md ¬ß4.2`](planner-harness.md)) removes
shell execution from the Reviewer role entirely. Tests, lint,
type-checking, and any other code-running verification can no longer
happen inside the Reviewer VM. The Reviewer reads code as text;
binary pass/fail and structured counters from code-running checks
must reach the Reviewer through a different channel.

That channel is the **mechanical witness flow** specified here. Every
witness ‚ÄĒ whether produced by an operator-policy claim-based gate,
a plan-author per-task verifier, or a plan-author pre-merge verifier
‚ÄĒ is generated by the same `raxis-verifier` binary running PID-1
in a VM, isolated from agent VMs, network-air-gapped by default,
producing a single bincode `WitnessSubmission` frame to the kernel
over VSock. Three distinct lifecycle hooks decide *when* the kernel
spawns these verifiers; the runtime is one.

Four properties this spec guarantees:

1. **Isolation of the verifier from the host and from agent VMs.**
   Verifier VMs have no LLM, no harness, no `IntentKind` dispatch
   ‚ÄĒ their only outbound channel is one `WitnessSubmission` frame.
   They are not observable by Executors, peer Reviewers, or the
   Orchestrator (`INV-VERIFIER-02`).

2. **Three authoring sources, one runtime.** Plan-author per-task
   verifiers (`[[plan.tasks.X.verifiers]]`), plan-author pre-merge
   verifiers (`[[plan.integration_merge_verifiers]]`), and
   operator-policy claim-based gates (`policy.toml [[gates]]` +
   `[[claim_requirements.rules]]`) all flow through the same
   verifier-VM scheduler. The kernel decides *when* to spawn them
   based on the lifecycle hook; the *how* is identical.

3. **Pre-staged artifacts for the Reviewer.** Symbol indexes,
   pre-rendered API documentation, JSON-formatted lint reports ‚ÄĒ
   content the Reviewer reads via `read_file` rather than running
   code to produce. Verifiers stage these into the dependent
   Reviewer's `/raxis/` mount per ¬ß6.

4. **Plan-author-controlled `on_failure` semantics.** Per-verifier
   failure routing ‚ÄĒ `block_review`, `block_merge`, or `warn_only`
   ‚ÄĒ selected at authoring time per ¬ß5.

## ¬ß2 ‚ÄĒ The Three Authoring Sources

All three produce structurally identical `witness_records` rows. They
differ in *who declares them* (policy author vs plan author) and *when
they fire* (which lifecycle hook).

| Source | Declared in | Lifecycle hook | Selection | Failure routing | Authority |
|---|---|---|---|---|---|
| **Policy claim-based gates** | `policy.toml [[gates]]` + `[[claim_requirements.rules]]` | `CompleteTask` admission | Kernel-derived from VCS diff (`INV-07`); the kernel reads `[[claim_requirements.rules]]` and decides which gates the diff demands | All required gates must pass; missing or failing witness blocks `CompleteTask` admission with `FAIL_INSUFFICIENT_WITNESS` | Operator (signed in policy bundle) |
| **Plan-author per-task verifiers** | `plan.toml [[plan.tasks.<id>.verifiers]]` | `CompleteTask` admission (after policy gates pass) | Plan-author, declared inline | Per-verifier `on_failure ‚ąą {block_review, warn_only}`; failures block Reviewer activation OR surface as flagged witness | Plan author (operator-signed plan bundle) |
| **Plan-author pre-merge verifiers** | `plan.toml [[plan.integration_merge_verifiers]]` and `policy.toml [[integration_merge_verifiers]]` | `IntegrationMerge` admission (Check 5d) | Plan-author or operator, with `applies_to ‚ąą {all, task_set, last}` scope filter | Per-verifier `on_failure ‚ąą {block_merge, warn_only}`; failures block the merge advance | Plan author OR operator; operator declarations cannot be downgraded |

**Why three sources, not one.** Claim-based gates protect operator
organizational invariants (you cannot ship a build that doesn't
compile because the operator says so); per-task verifiers implement
plan-author intent (this task should run integration tests because
the plan author says so); pre-merge verifiers validate the merged
state that no per-task verifier sees (cross-task interactions,
deploy smoke tests, end-to-end integration). Conflating any two
makes operator governance leaky (plan authors could weaken claim
coverage) or operator policy too rigid (every task gets every
operator-declared gate even when the plan author knows it's not
relevant).

This separation mirrors [`policy-plan-authority.md`](policy-plan-authority.md)'s broader
authority hierarchy: policy is operator-governed and signed by the
authority key; plan is plan-author-governed and signed by the
operator key for plan submission. The verifier layering inherits
this split.

**Where each source's failure code routes:**

```text
                         CompleteTask admission                    IntegrationMerge admission
                         ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ                     ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ

                                                                     (block_merge only)
```

## ¬ß3 ‚ÄĒ Per-Task Verifier Plan Declaration

```toml
[[plan.tasks.web_implementer.verifiers]]
name        = "node_test"            # required; unique within the task
image       = "raxis-verifier-node-starter"   # required; resolves to a [[vm_images]] entry whose role_restriction includes "Verifier"
command     = "/usr/local/bin/raxis-node-test" # required; single absolute check executable path run by raxis-verifier
timeout     = "10m"                  # required; wall-clock; kernel-enforced via cgroup.kill
on_failure  = "block_review"         # required; "block_review" | "warn_only"
artifact    = "/raxis/test_report.json"   # optional; see ¬ß6
artifact_max_bytes = 1048576         # optional; max staged-artifact size; default 1 MiB; max 64 MiB
env         = { CI = "true" }        # optional; merged with envelope env; cannot override RAXIS_*
```

**Field semantics:**

| Field | Required? | Purpose | Validation at `approve_plan` |
|---|---|---|---|
| `name` | Required | Identifier within the task; used for `verifier_witnesses` row keying and audit events | Non-empty; `[a-z][a-z0-9_]{0,31}`; unique within `[[verifiers]]` of this task |
| `image` | Required | OCI image used to boot the verifier VM | Must resolve to a `[[vm_images]]` entry whose `role_restriction` includes `"Verifier"` (per `INV-VM-CAP-03`); image must satisfy `INV-VERIFIER-06` (Linux 5.14+, cgroup v2, controllers in `subtree_control`) |
| `command` | Required | Single trusted mechanical check executable path run by the canonical `raxis-verifier` shim | Non-empty; absolute path; no whitespace; arguments and env belong inside the check wrapper executable |
| `timeout` | Required | Wall-clock timeout, parsed as a duration string ("30s", "10m", "1h") | Parses to ‚Č• 5 seconds and ‚Č§ kernel hard cap (`max_verifier_timeout_seconds` from `[host_capacity]`, default 1800 / 30 minutes) |
| `on_failure` | Required | Routing when the verifier reports `Fail`, times out, crashes, or fails artifact validation | `block_review` \| `warn_only`; per ¬ß5 |
| `artifact` | Optional | Absolute path inside the verifier VM whose contents are copied into the dependent Reviewer's `/raxis/` mount on success | If present: must start with `/raxis/`; max 256 chars; per ¬ß6 |
| `artifact_max_bytes` | Optional | Cap on staged artifact size | Default 1 MiB; max 64 MiB; rejected if > kernel-hard-cap `max_artifact_bytes` |
| `env` | Optional | Additional environment variables for the verifier process | `RAXIS_*` keys forbidden (collision with envelope); plain key=value strings only; max 32 entries; max 16 KiB total |

**`on_failure` is required, not optional.** Plan authors must
affirmatively choose between `block_review` and `warn_only` for
every per-task verifier. There is no kernel default. Silent defaults
for "what happens when this fails" are a known source of plan
authoring bugs; making the choice explicit forces consideration at
authoring time.

**No back-references between verifiers.** Per-task verifiers are
independent; there is no DAG or dependency declaration among them.
They run in parallel (subject to capacity caps); the kernel waits
for all of them to complete before evaluating Reviewer activation.
If a future need emerges for sequential verifiers (e.g., "build
first, then test against the build artifact"), that is addressed
via a single trusted check wrapper executable that performs those
steps, not by adding inter-verifier dependencies or shell command
strings. That wrapper does not speak the RAXIS witness protocol; the
kernel-spawned `raxis-verifier` shim owns token use, witness
submission, and witness acknowledgement.

## ¬ß4 ‚ÄĒ Verifier VM Lifecycle

Verifier VMs are kernel-spawned, single-purpose, short-lived. They
have no LLM, no harness, no inference. Their PID 1 is `raxis-verifier`
(`INV-VERIFIER-08`).

### 4.1 Activation Triggers

A verifier VM activates from one of three lifecycle hooks:

1. **Per-task verifier activation** ‚ÄĒ see ¬ß4.1.1.
2. **Pre-merge verifier activation** ‚ÄĒ see ¬ß4.1.2.
3. **Policy claim-based gate activation** ‚ÄĒ see ¬ß4.1.3 (the
   kernel-derived flow that runs at `CompleteTask` admission before
   per-task verifiers).

The spawn, execution, witness-submission, and teardown sequences
(¬ß4.2 through ¬ß4.4) are identical across all three; only the
trigger differs.

#### 4.1.1 Per-task verifier activation

Triggered when a task's Executor (or the task itself, for
non-Executor tasks that declare verifiers) submits
`IntentKind::CompleteTask` and the kernel admits the commit through
the standard admission pipeline. Specifically:

1. Executor submits `CompleteTask { commit_sha }`.
2. Kernel runs the standard `CompleteTask` admission checks
   (path allowlist, hybrid effective allow, claim_requirements
   policy gate evaluation ‚ÄĒ see ¬ß4.1.3 below).
3. **If policy claim-based gates have not yet completed:** the kernel
   returns `FAIL_INSUFFICIENT_WITNESS`; no per-task verifiers
   activate. The Executor waits and retries per ¬ß4.1.3.
4. **If policy claim-based gates have all passed (or there are none
   required):** the `CompleteTask` is admitted, `evaluation_sha` is
   set on the task, and the kernel reads the task's
   `[[plan.tasks.X.verifiers]]` declarations.
5. For each declared per-task verifier, the kernel allocates a
   verifier VM slot (against `[host_capacity] max_concurrent_verifier_vms`,
   default 8) and proceeds to ¬ß4.2. If allocation cannot complete
   within the admission queue policy from [`host-capacity.md ¬ß10`](host-capacity.md),
   verifiers queue.
6. Reviewer activation for the task is BLOCKED until all per-task
   verifiers complete (`final_status` in {`passed`, `failed`,
   `timed_out`, `crashed`, `artifact_missing`}). See ¬ß5 for the
   gating rules.

**Sequencing relative to policy claim-based gates.** Policy
claim-based gates run first; per-task verifiers run after they
pass. This ensures verifiers (which cost real resources) are not
consumed by tasks that fail kernel-derived gate evaluation.

#### 4.1.2 Pre-merge verifier activation

Triggered when the Orchestrator submits
`IntentKind::IntegrationMerge` (per [`integration-merge.md ¬ß2`](integration-merge.md)). The
admission pipeline reaches **Check 5d ‚ÄĒ Pre-Integration Merge
Verifier Execution** (per [`integration-merge.md ¬ß4 Check 5d`](integration-merge.md)):

1. Kernel computes the **candidate merged tree** as an orphan commit
   ([`integration-merge.md ¬ß4 Check 5d.2`](integration-merge.md)). The candidate SHA is
   ephemeral ‚ÄĒ it is discarded if verification fails (no main
   pollution) and renamed-and-pushed if verification passes.
2. Kernel reads:
   - `policy.toml [[integration_merge_verifiers]]` ‚ÄĒ operator-global
     pre-merge gates (cannot be downgraded by plan).
   - `plan.toml [[plan.integration_merge_verifiers]]` ‚ÄĒ plan-author
     pre-merge verifiers.
3. For each declared verifier whose `applies_to` matches this merge
   (per ¬ß16.3), the kernel allocates a verifier VM slot.
4. Verifier VMs are spawned with `/workspace` mounted from the
   candidate merged tree (NOT from any individual task branch and
   NOT from current main).
5. IntegrationMerge admission BLOCKS until all matching pre-merge
   verifiers complete.
6. Gating algorithm per ¬ß16.4: any `block_merge` failure ‚Üí kernel
   discards the candidate tree, returns
   `FAIL_INTEGRATION_MERGE_VERIFIER_BLOCKED { verifier_names,
   primary_witness_summary }` to the Orchestrator. All `block_merge`
   pass ‚Üí kernel mechanically advances main to the candidate SHA;
   IntegrationMerge succeeds.

#### 4.1.3 Policy claim-based gate activation

Triggered at `CompleteTask` admission (before per-task verifiers,
per ¬ß4.1.1 step 2). The kernel:

1. Computes the diff (`base_sha ‚Üí commit_sha`) and the touched-paths
   set.
2. Walks `policy.toml [[claim_requirements.rules]]` and determines
   which `[[gates]]` are required for this commit (e.g., "any path
   matching `*.rs` requires `RustBuild_Linux`").
3. For each required gate, queries `witness_records` for an existing
   `passed` witness against this `evaluation_sha`.
4. **If a required witness is missing:** the kernel spawns the gate's
   verifier VM (same scheduler as ¬ß4.1.1 / ¬ß4.1.2 ‚ÄĒ single runtime).
5. **If a required witness exists and is `failed` / `timed_out` /
   `crashed`:** kernel returns `FAIL_INSUFFICIENT_WITNESS` to the
   Executor. Executor revises and re-submits.
6. **If all required gates have passing witnesses:** admission
   continues to ¬ß4.1.1 step 4 (per-task verifier activation).

Gates' verifier VMs run via the same `raxis-verifier` PID-1 binary
on the same VM substrate as per-task and pre-merge verifiers. There
is no separate "host subprocess" execution path.

### 4.2 Spawn

Per verifier VM (identical for all three hook types):

1. **Worktree provisioning.**
   - **Per-task verifier:** kernel creates a fresh ephemeral clone of
     `main_repo @ evaluation_sha` at `worktrees/<verifier_run_id>/`.
     Clone is shallow (depth 1).
   - **Pre-merge verifier:** kernel creates a fresh ephemeral clone of
     the **candidate merged tree** at `worktrees/<verifier_run_id>/`.
     The candidate SHA is the orphan commit produced by
     [`integration-merge.md ¬ß4 Check 5d.2`](integration-merge.md).
   - **Policy claim-based gate:** same as per-task verifier ‚ÄĒ
     `main_repo @ evaluation_sha`.
2. **VM allocation.** Kernel boots the VM from the image declared by
   `image` (digest-verified per `INV-VM-CAP-03`). VM gets:
   - `/workspace` mounted RW from the ephemeral clone (VirtioFS)
   - `/raxis/` mounted RW (verifier-only; written artifacts staged
     from here into the dependent consumer's `/raxis/`)
   - cgroup v2 substrate per `INV-VERIFIER-06`
   - VSock for witness submission to the kernel
   - **No** network interface (no tproxy, no Credential Proxy);
     verifier VMs are network-isolated (`INV-VERIFIER-11`). If a
     verifier genuinely needs network access (rare; e.g., fetching a
     fixture), the operator declares it via
     `[[plan.tasks.X.verifiers.allowed_egress]]` (per-task) or
     `[[plan.integration_merge_verifiers.allowed_egress]]`
     (pre-merge). **Default is no network.**
3. **Envelope.** Kernel writes the verifier-spawn envelope:
   - Common: `RAXIS_VERIFIER_TOKEN`, `RAXIS_TASK_ID` (or
     `RAXIS_INTEGRATION_MERGE_ID` for pre-merge),
     `RAXIS_EVALUATION_SHA`, `RAXIS_KERNEL_SOCKET` (per-VM VSock
     port), `RAXIS_INITIATIVE_ID`
   - `RAXIS_VERIFIER_NAME` ‚ÄĒ the `name` from the declaration
   - `RAXIS_VERIFIER_COMMAND` ‚ÄĒ the `command` to execute
   - `RAXIS_VERIFIER_TIMEOUT_MS` ‚ÄĒ the parsed `timeout` in ms
   - `RAXIS_VERIFIER_ARTIFACT_PATH` ‚ÄĒ the optional `artifact` path
     (or empty string)
   - `RAXIS_VERIFIER_ARTIFACT_MAX_BYTES` ‚ÄĒ declared cap on artifact
     size (or 0 if no artifact)
   - `RAXIS_VERIFIER_HOOK_KIND` ‚ÄĒ `"per_task"` | `"pre_merge"` |
     `"policy_claim_gate"` (so the verifier command can adapt if it
     wants; rarely used)
   - Plus operator-declared `env` entries from ¬ß3
4. **Cgroup setup.** Kernel writes the VM's QEMU/Firecracker process
   to `/sys/fs/cgroup/raxis/verifier-<verifier_run_id>/` on the
   host with `memory.max` and `cpu.max` derived from policy
   (`max_verifier_memory_mb`, `verifier_cpu_quota`). In-VM,
   `raxis-verifier` sets up its own internal cgroups for the spawned
   verifier executable per `INV-PLANNER-HARNESS-03`.
5. **Audit emission.** `VerifierActivated { hook_kind, task_id (or
   integration_merge_id), verifier_name, verifier_run_id,
   image_digest, evaluation_sha (or candidate_merge_sha for
   pre-merge), command, timeout_ms, on_failure_action }` audit
   event written before VM start.
6. **VM start.** Kernel calls hypervisor to boot the VM. PID 1
   (`raxis-verifier`) reads its envelope and proceeds.

### 4.3 Execution (`raxis-verifier` PID-1 Behavior)

`raxis-verifier` is a small statically-linked binary (not the
planner, not claw-code-derived) whose entire job is to:

1. Read envelope (env vars).
2. Verify the envelope's required vars are present; abort with
   exit code 64 (`EX_USAGE`) and submit a `Crashed` witness if not.
3. Mount cgroup v2 sub-cgroups for the upcoming subprocess
   (`/sys/fs/cgroup/raxis/verifier-cmd/`, `cpu.weight = 100`).
4. Fork+exec the single absolute `RAXIS_VERIFIER_COMMAND` check path with
   stdout and stderr piped, working directory set to `/workspace`,
   environment = envelope ‚ą™ operator `env`. The subprocess is placed
   in the sub-cgroup before exec. If the verifier needs arguments or
   multi-step behavior, those belong inside the trusted check wrapper
   executable. The check process must not submit witnesses directly;
   it exits with process status and lets `raxis-verifier` submit the
   mechanical witness.
5. Concurrently:
   - Drain stdout into a ring buffer (last 4 KiB retained;
     configurable via `[host_capacity] verifier_stdout_tail_bytes`,
     default 4096, max 65536).
   - Drain stderr into a ring buffer (same bytes cap).
   - Attempt structured output detection on the streamed bytes:
     - JUnit XML on stdout ‚Üí parse `<testsuite tests="N"
       failures="M" errors="K">` aggregates
     - `cargo test --message-format=json` lines ‚Üí aggregate
       `test.passed`, `test.failed`, `test.ignored`
     - LCOV `tracefile` ‚Üí aggregate line/branch coverage if present
     - `--quiet` or arbitrary text ‚Üí no aggregates; counters are
       NULL
   - Wait for subprocess exit OR the wall-clock timeout
     (`RAXIS_VERIFIER_TIMEOUT_MS`).
6. **On timeout:** write `"1"` to
   `/sys/fs/cgroup/raxis/verifier-cmd/cgroup.kill`, wait for
   `cgroup.events` `populated=0`, capture exit signal as `SIGKILL`,
   set `final_status = "timed_out"`, and proceed to step 7.
7. **Artifact staging:** if `RAXIS_VERIFIER_ARTIFACT_PATH` is
   non-empty:
   - Verify file exists at the declared path.
   - Verify file is non-empty.
   - Verify file size ‚Č§ `RAXIS_VERIFIER_ARTIFACT_MAX_BYTES`.
   - On any failure: set `final_status = "artifact_missing"`
     (overrides prior status of `passed`; does NOT override
     `failed`, `timed_out`, or `crashed`).
   - On success: the artifact bytes are part of the witness record
     submission (kernel will copy them to its staging directory).
8. **Witness submission:** open `RAXIS_KERNEL_SOCKET` (VSock) and
   submit one `WitnessSubmission` frame (bincode, length-prefixed;
   see ¬ß7 for schema).
9. **Exit:** exit with code 0 after witness submission attempted.
   The kernel reads `final_status` from the witness body, not the
   verifier exit code.

If `raxis-verifier` itself crashes (segfault, OOM-killed by host
cgroup, panic before witness submission), the kernel detects this
when the VM exits without a corresponding witness. It synthesizes
a `Crashed` witness (`final_status = "crashed"`) and emits
`VerifierVmCrashed` audit. The verifier run does NOT silently
disappear.

### 4.4 Teardown

After witness submission (or crash detection):

1. Kernel emits the appropriate audit event from ¬ß11.
2. Kernel persists the witness row (per ¬ß7) within a single SQLite
   transaction.
3. Kernel writes any staged artifact bytes to the per-session
   staging directory:
   - Per-task verifier: `staging/<task_id>/<verifier_name>/<artifact_basename>`
   - Pre-merge verifier: `staging/merge/<integration_merge_id>/<verifier_name>/<artifact_basename>`
   - Policy claim-based gate: `staging/<task_id>/<verifier_name>/<artifact_basename>`
   Atomically (write to `<artifact_basename>.tmp`, fsync, rename).
4. Kernel calls hypervisor to stop the VM.
5. Kernel removes the host-side cgroup
   `verifier-<verifier_run_id>/` after `cgroup.procs` is empty.
6. Kernel removes the ephemeral worktree clone at
   `worktrees/<verifier_run_id>/`.

The teardown sequence is idempotent. If the kernel restarts
mid-teardown (e.g., between step 2 and step 4), the startup
recovery path ([`kernel-lifecycle.md ¬ß7`](kernel-lifecycle.md)) detects orphaned VMs by
their cgroup presence and finishes their teardown before resuming
intent admission.

## ¬ß5 ‚ÄĒ `on_failure` Semantics and Activation Gating

Three valid `on_failure` values exist across the two plan-author
authoring sources (per-task and pre-merge); operator policy claim
gates have implicit `block_admit` semantics that don't use this
field.

### 5.1 The Three Modes

| Value | Used by | Semantics on non-passing `final_status` | Audit emitted |
|---|---|---|---|
| `block_review` | Per-task verifiers only | Dependent Reviewer NEVER activates for this `evaluation_sha`. Originating Executor's `CompleteTask` is rolled into `Failed` per [`agent-disagreement.md ¬ß3`](agent-disagreement.md). Executor receives `FAIL_VERIFIER_BLOCKED` on its next intent. | `VerifierBlockedReview { task_id, verifier_name, final_status }` plus per-status event from ¬ß11 |
| `block_merge` | Pre-merge verifiers only | Candidate merged tree is discarded. Orchestrator receives `FAIL_INTEGRATION_MERGE_VERIFIER_BLOCKED { verifier_names, primary_witness_summary }` and routes per [`agent-disagreement.md ¬ß6`](agent-disagreement.md) (typically: escalate to operator). Main is NOT advanced. | `VerifierBlockedMerge { integration_merge_id, verifier_names, candidate_merge_sha }` plus per-status event |
| `warn_only` | Per-task AND pre-merge verifiers | The dependent consumer (Reviewer or merge-advance, respectively) IS activated/advanced normally. Witness is surfaced in the consumer's KSB / audit with a flag indicating the warning. | Per-status event from ¬ß11 only; no `VerifierBlockedReview` / `VerifierBlockedMerge` |

**Authoring-time rules:**

- A per-task verifier with `on_failure = "block_merge"` is
  **rejected at admission** with `FAIL_VERIFIER_INVALID_ON_FAILURE
  { verifier_name, declared: "block_merge", allowed: ["block_review",
  "warn_only"] }`. Per-task verifiers cannot block merges; their
  scope is per-task Reviewer activation.
- A pre-merge verifier with `on_failure = "block_review"` is
  similarly rejected. Pre-merge verifiers cannot block Reviewer
  activation; the Reviewer for the contributing tasks already ran
  before pre-merge verification fires.
- Operator policy claim-based gates do not declare `on_failure` ‚ÄĒ
  their semantics are implicit (`block_admit` at `CompleteTask`).

### 5.2 Reviewer Activation Gating Algorithm (per-task verifiers)

After a task's per-task verifiers have all completed (every verifier
declared in `[[plan.tasks.X.verifiers]]` has a witness row with a
non-NULL `final_status`):

```yaml
let block_review_failures = task.verifiers.iter()
    .filter(|v| v.on_failure == "block_review" && v.final_status != "passed")
    .collect::<Vec<_>>();

if !block_review_failures.is_empty() {
    // Roll Executor's CompleteTask into Failed
    transition_task_to_failed(
        task_id,
        FailureReason::VerifierBlockedReview {
            verifier_names: block_review_failures.iter().map(|v| v.name.clone()).collect(),
            primary_witness_summary: block_review_failures[0].summary(),
        },
    );
    // Reviewer is never activated for this evaluation_sha.
    // The Executor receives FAIL_VERIFIER_BLOCKED on its next intent and
    // can choose to revise (per agent-disagreement.md ¬ß2 max_review_rounds
    // semantics ‚ÄĒ verifier failure counts as a review round for cap purposes).
} else {
    // All block_review verifiers passed (or there are none).
    // Activate Reviewer with the full witness set in its KSB.
    activate_reviewer(task_id, evaluation_sha, build_verifier_witnesses_section(...));
}
```

**`block_review` failures count as review rounds** for the purpose
of [`agent-disagreement.md ¬ß2`](agent-disagreement.md) `INV-CONVERGENCE-01`. This prevents
an Executor from looping indefinitely on a verifier failure (e.g.,
the build is genuinely broken in a way the LLM can't fix;
`max_review_rounds` fires, the task escalates per ¬ß6 of
[`agent-disagreement.md`](agent-disagreement.md)).

### 5.3 IntegrationMerge Advance Gating Algorithm (pre-merge verifiers)

After all matching pre-merge verifiers for an `IntegrationMerge`
attempt have completed, the algorithm runs at `integration-merge.md
¬ß4 Check 5d` admission:

```yaml
let block_merge_failures = pre_merge_verifiers.iter()
    .filter(|v| applies_to_matches(v, current_merge))
    .filter(|v| v.on_failure == "block_merge" && v.final_status != "passed")
    .collect::<Vec<_>>();

if !block_merge_failures.is_empty() {
    // Discard the candidate merged tree.
    discard_candidate_merge_tree(candidate_merge_sha);
    // Return failure to Orchestrator.
    return FailureReason::IntegrationMergeVerifierBlocked {
        verifier_names: block_merge_failures.iter().map(|v| v.name.clone()).collect(),
        primary_witness_summary: block_merge_failures[0].summary(),
        candidate_merge_sha,
    };
    // Main is NOT advanced. Orchestrator receives the failure and
    // routes per agent-disagreement.md ¬ß6 (typically operator escalation).
} else {
    // All block_merge verifiers passed (or there are none matching this merge).
    // Advance main to candidate_merge_sha; IntegrationMerge succeeds.
    proceed_to_check_6_and_8(candidate_merge_sha);
}
```

**Pre-merge verifier failures do NOT count toward
`INV-CONVERGENCE-01`** because they are not review rounds ‚ÄĒ the
per-task Reviewers already approved the contributing tasks. A
pre-merge failure is a higher-order regression; the Orchestrator
typically cannot self-resolve it and must escalate to the operator
per [`agent-disagreement.md ¬ß6`](agent-disagreement.md).

### 5.4 Edge Cases

- **No verifiers declared.** Reviewer is activated immediately
  after `CompleteTask` admission (no per-task verifiers); merge is
  advanced immediately after Check 5c (no pre-merge verifiers).
  Empty `verifier_witnesses` section in the KSB / audit. Valid
  plan-author choice; the kernel does not require either.
- **Mixed `on_failure` modes within a hook.** Each verifier is
  evaluated independently. A task with one `block_review` failure
  and three `warn_only` passes does NOT activate the Reviewer
  (the block_review failure dominates). Same logic for `block_merge`
  in the pre-merge hook.
- **Verifier crashes before submitting witness.** Treated as
  `final_status = "crashed"` (synthesized by kernel on VM exit
  without witness). For `block_review` / `block_merge`, this blocks
  the dependent operation. For `warn_only`, the consumer sees a
  `crashed` witness.
- **Plan declares a verifier whose image is no longer in
  `[[vm_images]]`.** Detected at `approve_plan` time (per ¬ß3
  validation table); plan is rejected before activation. Cannot
  occur at runtime.
- **Operator amends policy bundle to remove the verifier image
  mid-initiative.** Detected at verifier-activation time; kernel
  emits `VerifierImageDigestUnavailable` and treats the verifier as
  crashed per `on_failure`. Operator must amend the plan or revert
  the policy change.

## ¬ß6 ‚ÄĒ The `artifact` Mechanism

Verifiers may pre-stage files into the dependent consumer's `/raxis/`
mount, available via `read_file`. The dependent consumer is:

- **Per-task verifier:** the Reviewer for that task.
- **Pre-merge verifier:** Future Reviewers in subsequent merges
  within the same initiative MAY consume artifacts staged at
  `staging/merge/<integration_merge_id>/...`, but this is opt-in
  per the consumer's `read_file` access; the artifacts are NOT
  auto-mounted into per-task Reviewer VMs.
- **Policy claim-based gate:** the Reviewer for the task whose
  `CompleteTask` triggered gate evaluation (same as per-task).

### 6.1 Use Cases

- **Symbol indexes** (`/raxis/symbol_index.json`) produced by a
  parser-only verifier (e.g., wrapping `ctags`); Reviewer reads to
  resolve symbols without having an LSP (per `planner-harness.md
  ¬ß4.1` operator note). The kernel-canonical
  `raxis-verifier-symbol-index` image (¬ß14) is the canonical
  producer.
- **Pre-rendered API documentation** (`/raxis/api_docs.html` or
  `.md`) produced by `cargo doc`, `typedoc`, etc.
- **JSON-formatted lint reports** (`/raxis/clippy.json`,
  `/raxis/eslint.json`).
- **Coverage maps** (`/raxis/coverage.lcov`) for Reviewer to see
  which lines of the diff are exercised by tests.
- **Dependency graphs**, **AST dumps**, **call graphs** ‚ÄĒ any
  Reviewer-consumable structured output.

### 6.2 Artifact Lifecycle

1. **Verifier writes** to `/raxis/<artifact_path>` inside its VM
   during the command's execution.
2. **`raxis-verifier` validates** at exit (per ¬ß4.3 step 7): file
   exists, non-empty, ‚Č§ `artifact_max_bytes`.
3. **Witness submission** includes the artifact bytes in the
   `WitnessSubmission` frame (per ¬ß7).
4. **Kernel writes** the artifact to per-session staging atomically
   (tmp + fsync + rename); path per ¬ß4.4 step 3.
5. **Reviewer activation:** kernel mounts `/raxis/` (RO from the
   Reviewer's perspective) into the Reviewer VM, with all staged
   artifacts present at `/raxis/<verifier_name>/<artifact_basename>`.
   The Reviewer reads them via `read_file`.
6. **Cleanup:** staging directory is removed when the task (or the
   IntegrationMerge attempt) transitions to a terminal state
   (`Done`, `Failed`, `Aborted`).

### 6.3 Validation Failures

| Condition | `final_status` | Audit | Effect |
|---|---|---|---|
| Verifier exit 0, artifact path missing | `artifact_missing` | `VerifierArtifactMissing` | Per `on_failure` |
| Verifier exit 0, artifact present but empty | `artifact_missing` | Same | Same |
| Verifier exit 0, artifact present but > `artifact_max_bytes` | `artifact_missing` | `VerifierArtifactOversize` | Same |
| Verifier exit 0, artifact valid | `passed` | `VerifierCompleted` | Consumer activated/advanced with artifact staged |
| Verifier exit ‚Č† 0 | `failed` | `VerifierCompleted` | Per `on_failure` (artifact validation skipped) |

**Rationale for treating oversize as missing:** `artifact_max_bytes`
is a plan-declared cap. Exceeding it means the verifier produced
something the plan author did not authorize for staging. Cleaner
to treat as missing and let the plan author either widen the cap
or fix the verifier than to partial-stage with truncation
sentinels.

### 6.4 Artifact Scoping

Artifacts are staged into the dependent consumer's namespace
(per-task or per-merge-attempt). They are NOT visible to peer
Executors, sibling tasks, the Orchestrator (other than via the
candidate-merge-tree itself), or operations across the
peer-task / peer-merge boundary.

If a future need emerges for cross-task artifact sharing, it will
be addressed via a separate mechanism with its own audit
semantics, not by widening the artifact mechanism's scope.

## ¬ß7 ‚ÄĒ `WitnessSubmission` Schema

There is one schema. All three authoring sources produce the same
frame; the kernel routes it based on the verifier's lifecycle hook
(recorded at spawn time, looked up by `verifier_run_id`).

```rust
pub struct WitnessSubmission {
    pub verifier_token:      String,            // shared envelope mechanic
    pub verifier_run_id:     VerifierRunId,
    pub verifier_name:       String,            // matches the declaration's `name`
    pub evaluation_sha:      String,            // task evaluation_sha or candidate_merge_sha
    pub final_status:        FinalStatus,
    pub exit_code:           Option<i32>,       // None if timed_out / crashed
    pub runtime_ms:          u64,
    pub stdout_tail:         Vec<u8>,           // last N bytes
    pub stderr_tail:         Vec<u8>,           // last N bytes
    pub structured_counters: Option<Json>,      // detected JUnit / cargo-test / LCOV aggregates
    pub artifact:            Option<Artifact>,
}

pub enum FinalStatus {
    Passed,
    Failed,
    TimedOut,
    Crashed,
    ArtifactMissing,
}

pub struct Artifact {
    pub relative_path: String,                  // basename only; e.g., "symbol_index.json"
    pub bytes:         Vec<u8>,                 // capped at artifact_max_bytes
    pub sha256:        [u8; 32],                // computed by raxis-verifier; kernel verifies
}
```

Bincode framing per `peripherals.md ¬ß3.3`.

### 7.1 `witness_records` Schema

One table, one schema, no `witness_version` discriminator. The
kernel knows which lifecycle hook produced each witness via the
`hook_kind` column populated at spawn time (not from the witness
payload ‚ÄĒ the verifier doesn't need to know how it was triggered
for routing purposes).

```sql
CREATE TABLE witness_records (
    verifier_run_id        TEXT PRIMARY KEY,
    verifier_run_token_id  TEXT NOT NULL REFERENCES verifier_run_tokens(id),
    initiative_id          TEXT NOT NULL,
    hook_kind              TEXT NOT NULL,        -- 'per_task' | 'pre_merge' | 'policy_claim_gate'
    task_id                TEXT,                 -- non-NULL for per_task and policy_claim_gate
    integration_merge_id   TEXT,                 -- non-NULL for pre_merge
    verifier_name          TEXT NOT NULL,
    evaluation_sha         TEXT NOT NULL,
    final_status           TEXT NOT NULL,         -- 'passed' | 'failed' | 'timed_out' | 'crashed' | 'artifact_missing'
    exit_code              INTEGER,
    runtime_ms             INTEGER NOT NULL,
    stdout_tail            BLOB,
    stderr_tail            BLOB,
    structured_counters    TEXT,                  -- JSON
    artifact_relative_path TEXT,
    artifact_sha256        TEXT,                  -- hex
    artifact_size_bytes    INTEGER,
    on_failure_action      TEXT,                  -- 'block_review' | 'block_merge' | 'warn_only' | 'block_admit'
    image_digest           TEXT NOT NULL,
    created_at             INTEGER NOT NULL       -- epoch ms
);

CREATE INDEX idx_witness_records_per_task ON witness_records (task_id, verifier_name)
    WHERE hook_kind IN ('per_task', 'policy_claim_gate');
CREATE INDEX idx_witness_records_pre_merge ON witness_records (integration_merge_id, verifier_name)
    WHERE hook_kind = 'pre_merge';
CREATE INDEX idx_witness_records_initiative ON witness_records (initiative_id);
```

The `verifier_run_id` PK and FK to `verifier_run_tokens` are the
authentication mechanism; tokens are issued at spawn time per the
existing token-issuance flow (carried over conceptually from V1
with no semantic change).

## ¬ß8 ‚ÄĒ Reviewer KSB Integration

The Reviewer's KSB (per [`kernel-mechanics-prompt.md`](kernel-mechanics-prompt.md)) carries a
first-class `verifier_witnesses` section listing every per-task
verifier (and every policy-claim-based gate witness) for the
`evaluation_sha` under review. **Pre-merge verifiers are NOT
surfaced in the per-task Reviewer's KSB** ‚ÄĒ they fire after the
Reviewer has already acted; the Orchestrator sees them.

Rendering shape (pseudo):

```text
Verifier Witnesses (for evaluation_sha = abc123def...):

  ‚úď rust_build (passed in 47.4s)            [policy_claim_gate]
    exit 0

  ‚úď unit_test (passed in 12.4s)             [per_task]
    exit 0, 142 tests passed, 0 failed (cargo test --message-format=json)

  ‚úď symbol_index (passed in 0.8s)           [per_task]
    exit 0, artifact staged at /raxis/symbol_index/symbol_index.json (812 KiB)

  ‚ö† integration_test (failed in 4m31s; on_failure = warn_only)   [per_task]
    exit 1, last stderr:
    ‚ĒÉ Error: API key for staging environment expired
    ‚ĒÉ Test 'authenticated_endpoints' failed: 401 Unauthorized
    Note: this verifier's failure does NOT block your review (warn_only).
    Decide whether to approve based on the failure context.
```

Rendering rules:

- One row per witness for this `evaluation_sha`, ordered: policy
  claim-gates first, then per-task verifiers (alphabetical by
  `name` within each group)
- `final_status` rendered as `‚úď passed` / `‚úó failed` / `‚ö† failed
  (warn_only)` / `‚ŹĪ timed_out` / `ūüí• crashed` / `ūüďĀ artifact_missing`
  (icons are stable conventions)
- For non-passing rows: `on_failure` mode displayed with a one-line
  interpretation hint (`block_review` failures don't appear here ‚ÄĒ
  those produce `FAIL_VERIFIER_BLOCKED` and no Reviewer activation)
- For passing rows with artifacts: artifact path inside the
  Reviewer VM and size

The full schema (field names, exact rendering rules, alert
priorities) is in [`kernel-mechanics-prompt.md`](kernel-mechanics-prompt.md).

**No `read_witness` tool.** The Reviewer doesn't choose which
witnesses to inspect; every witness for the Executor's
`evaluation_sha` is in the KSB by construction. This makes "the
Reviewer ignored the failing test" structurally impossible rather
than a vigilance question.

## ¬ß9 ‚ÄĒ Sequencing, Concurrency, and Capacity

### 9.1 Per-Task Sequence

```text
Executor commits (CompleteTask submitted)
  ‚Ēā
  ‚Ėľ
Policy claim-based gates evaluation
  ‚Ēú‚ĒÄ if any required gate Fail/Missing ‚Üí spawn gate verifier OR
  ‚Ēā    return FAIL_INSUFFICIENT_WITNESS to Executor; loop
  ‚ĒĒ‚ĒÄ if all pass ‚Üí proceed
  ‚Ēā
  ‚Ėľ
Per-task verifiers spawn (parallel, per host capacity)
  ‚Ēú‚ĒÄ Each VM: clone evaluation_sha, run command, capture, witness, exit
  ‚ĒĒ‚ĒÄ Kernel waits for all to complete (final_status set)
  ‚Ēā
  ‚Ėľ
Reviewer activation gating (¬ß5.2)
  ‚Ēú‚ĒÄ if any block_review verifier failed ‚Üí task ‚Üí Failed state
  ‚ĒĒ‚ĒÄ if all block_review pass ‚Üí activate Reviewer with verifier_witnesses KSB
```

### 9.2 Pre-Merge Sequence

```text
Orchestrator submits IntegrationMerge { commit_sha, merged_task_ids }
  ‚Ēā
  ‚Ēú‚ĒÄ V2.7 wire compatibility: if merged_task_ids is not present,
  ‚Ēā  kernel derives it as all completed Executor tasks and first
  ‚Ēā  verifies each Executor evaluation_sha is an ancestor of commit_sha
  ‚Ēā
  ‚Ėľ
integration-merge.md Check 1..5c (all pass)
  ‚Ēā
  ‚Ėľ
Check 5d ‚ÄĒ Pre-Integration Merge Verifier Execution (this spec)
  ‚Ēā
  ‚Ėľ
Compute candidate merged tree as orphan commit (candidate_merge_sha)
  ‚Ēā
  ‚Ėľ
Filter pre-merge verifiers by applies_to:
  - applies_to = "all"        ‚Üí always
  - applies_to = "task_set"   ‚Üí if merged_task_ids ‚ą© verifier.task_set ‚Č† ‚ąÖ
  - applies_to = "last"       ‚Üí only if this is the final merge of the DAG
  ‚Ēā
  ‚Ėľ
Spawn matching pre-merge verifiers (parallel)
  ‚Ēú‚ĒÄ Each VM: clone candidate_merge_sha, run command, capture, witness, exit
  ‚ĒĒ‚ĒÄ Kernel waits for all to complete
  ‚Ēā
  ‚Ėľ
IntegrationMerge advance gating (¬ß5.3)
  ‚Ēú‚ĒÄ if any block_merge verifier failed ‚Üí discard candidate; return
  ‚Ēā    FAIL_INTEGRATION_MERGE_VERIFIER_BLOCKED to Orchestrator
  ‚ĒĒ‚ĒÄ if all block_merge pass ‚Üí proceed to Check 6, advance main
```

### 9.3 Parallelism

Verifiers run **in parallel** within a single hook invocation by
default. There is no plan-level ordering hint. Operators wanting
strict ordering encode it inside the trusted verifier wrapper
executable itself.

The kernel-wide concurrency cap is `[host_capacity]
max_concurrent_verifier_vms` (default 8). When the cap is reached,
additional verifier VMs queue per the standard `host-capacity.md
¬ß10` admission queue policy (round-robin per initiative,
per-operator quotas honored).

**Hook-aware queueing.** Pre-merge verifiers do NOT preempt
per-task verifiers (or vice versa); they all share the same queue.
Operators with heavy pre-merge verification should size
`max_concurrent_verifier_vms` accordingly.

### 9.4 Capacity Composition

Verifier VMs consume the same host budget as planner VMs.
Specifically, each active verifier VM counts against:

- `max_concurrent_vms` (per [`host-capacity.md ¬ß4`](host-capacity.md)) ‚ÄĒ the global
  VM cap, shared with planner VMs
- `max_aggregate_vm_memory_mb` ‚ÄĒ verifier VMs' memory caps add to
  the aggregate
- `max_concurrent_verifier_vms` ‚ÄĒ sub-cap specifically for verifier
  VMs, default `min(8, max_concurrent_vms / 2)`

The verifier sub-cap exists to prevent a burst of completing tasks
(or a complex DAG with many parallel pre-merge verifiers) from
spawning enough verifier VMs to starve planner VMs. Operators
running heavy verifier workloads can raise it, accepting that
planner-VM availability shrinks correspondingly.

### 9.5 Per-Verifier Resource Caps

```toml
[host_capacity]
max_concurrent_verifier_vms  = 8                # global verifier-VM concurrency cap
max_verifier_memory_mb       = 4096             # per-verifier-VM memory cap; default 4 GiB
verifier_cpu_quota           = "1.0"            # cgroup cpu.max: 1.0 = 1 full vCPU
max_verifier_timeout_seconds = 1800             # hard kernel cap on declared per-verifier timeout; default 30 min
verifier_stdout_tail_bytes   = 4096             # bytes of stdout retained per verifier
verifier_stderr_tail_bytes   = 4096             # bytes of stderr retained per verifier
max_artifact_bytes           = 67108864         # hard kernel cap on per-artifact size; default 64 MiB
```

These are operator-policy defaults; per-verifier `timeout` and
`artifact_max_bytes` in `plan.toml` cannot exceed the corresponding
hard kernel caps.

## ¬ß10 ‚ÄĒ Failure Modes and Recovery

### 10.1 Verifier-Side Failures

| Failure mode | Detection | `final_status` | Effect with `block_review` / `block_merge` | Effect with `warn_only` |
|---|---|---|---|---|
| Command exit 0, no artifact declared | `raxis-verifier` exit | `passed` | Consumer activated/advanced | Consumer activated/advanced |
| Command exit 0, artifact valid | `raxis-verifier` exit | `passed` | Consumer activated/advanced, artifact staged | Consumer activated/advanced, artifact staged |
| Command exit 0, artifact missing/empty/oversize | `raxis-verifier` exit | `artifact_missing` | Consumer NOT activated/advanced; task ‚Üí Failed (per_task) or merge rejected (pre_merge) | Consumer activated/advanced; KSB/audit warning |
| Command exit ‚Č† 0 | `raxis-verifier` exit | `failed` | Consumer NOT activated/advanced | Consumer activated/advanced; KSB/audit warning |
| Command timed out | `raxis-verifier` cgroup.kill at deadline | `timed_out` | Consumer NOT activated/advanced | Consumer activated/advanced; KSB/audit warning |
| `raxis-verifier` itself crashed | Kernel detects VM exit without witness submission | `crashed` (synthesized) | Consumer NOT activated/advanced | Consumer activated/advanced; KSB/audit warning |
| VM crashed (hypervisor failure, host OOM-kill) | Kernel's hypervisor watchdog | `crashed` (synthesized) | Consumer NOT activated/advanced | Consumer activated/advanced; KSB/audit warning |
| Verifier image OCI digest unavailable | At spawn time | `crashed` (synthesized; no VM ever booted) | Consumer NOT activated/advanced | Consumer activated/advanced; KSB/audit warning |

### 10.2 Kernel-Side Failures

| Failure mode | Detection | Recovery |
|---|---|---|
| Kernel restart with verifier VM mid-execution | At startup, scan `/sys/fs/cgroup/raxis/verifier-*/` and `worktrees/<verifier_run_id>/` for orphans | Kill orphan VMs via `cgroup.kill`, synthesize `crashed` witness for each, treat per `on_failure` |
| Kernel restart between witness write and consumer activation | At startup, scan `tasks` and `integration_merge_attempts` tables for entries whose verifiers have completed but the gating algorithm has not run | Resume the appropriate gating algorithm (¬ß5.2 or ¬ß5.3) |
| Kernel restart between verifier completion and artifact staging | At startup, scan staging directory for partial writes (`.tmp` files) | Discard partial writes; if witness has artifact bytes, retry staging; if witness lost (race), synthesize `artifact_missing` |
| Kernel restart between artifact staging and consumer mount | Staging directory is intact at startup | Idempotent; gating proceeds with staged artifacts |
| Disk full during artifact staging | Kernel halt-admission policy from [`host-capacity.md ¬ß7`](host-capacity.md) | Verifier's witness is held; staging retried after disk recovery; if persistent, synthesize `artifact_missing` |
| Pre-merge verifier completes after kernel restart, but candidate merge tree was discarded | At startup, scan `integration_merge_attempts` for in-flight pre-merge runs; cross-check against `worktrees/` for the candidate clone | If candidate clone is intact, proceed; if discarded, synthesize all in-flight verifiers as `crashed` and re-spawn |

### 10.3 Operator-Side Failures (caught at `approve_plan`)

| Failure mode | When detected | Resolution |
|---|---|---|
| Plan declares verifier whose `image` is not in policy `[[vm_images]]` | `approve_plan` | `FAIL_VM_IMAGE_NOT_REGISTERED` |
| Image exists but `role_restriction` lacks `Verifier` | `approve_plan` | `FAIL_VM_IMAGE_ROLE_RESTRICTION_MISMATCH` |
| Plan declares `timeout` exceeding `max_verifier_timeout_seconds` | `approve_plan` | `FAIL_VERIFIER_TIMEOUT_EXCEEDS_HARD_CAP` |
| Plan declares `artifact_max_bytes` exceeding `max_artifact_bytes` | `approve_plan` | `FAIL_VERIFIER_ARTIFACT_CAP_EXCEEDS_HARD_CAP` |
| Plan declares `name` collision within same hook | `approve_plan` | `FAIL_VERIFIER_NAME_COLLISION` |
| Plan declares verifier with no `command` or empty `command` | `approve_plan` | `FAIL_VERIFIER_COMMAND_REQUIRED` |
| Per-task verifier with `on_failure = "block_merge"` | `approve_plan` | `FAIL_VERIFIER_INVALID_ON_FAILURE` |
| Pre-merge verifier with `on_failure = "block_review"` | `approve_plan` | `FAIL_VERIFIER_INVALID_ON_FAILURE` |
| Pre-merge verifier with `applies_to = "task_set"` and empty `task_set` | `approve_plan` | `FAIL_VERIFIER_TASK_SET_EMPTY` |
| Pre-merge verifier with `applies_to = "task_set"` referencing undeclared task ID | `approve_plan` | `FAIL_VERIFIER_TASK_SET_UNKNOWN_TASK` |

All `FAIL_*` codes here are home-cataloged in
[`policy-plan-authority.md ¬ß3b`](policy-plan-authority.md).

| `VerifierArtifactMissing` | `final_status = "artifact_missing"` | Same scope keys + `declared_artifact_path`, `observed_artifact_size_bytes` |
| `VerifierArtifactOversize` | Sub-event when file exists but exceeds `artifact_max_bytes` | Same + `artifact_max_bytes` |
| `VerifierBlockedReview` | A `block_review` verifier produced a non-passing `final_status` | `task_id`, `verifier_names: Vec<String>`, `primary_witness_summary` |
| `VerifierBlockedMerge` | A `block_merge` verifier produced a non-passing `final_status` | `integration_merge_id`, `candidate_merge_sha`, `verifier_names: Vec<String>`, `primary_witness_summary` |
| `VerifierImageDigestUnavailable` | At spawn time, the image is no longer in policy `[[vm_images]]` | Scope keys + `image_pinned_digest`, `policy_epoch_at_spawn` |
| `CandidateMergeTreeCreated` | Pre-merge verifier hook computed the candidate merge | `integration_merge_id`, `candidate_merge_sha`, `merged_task_ids` |
| `CandidateMergeTreeDiscarded` | Pre-merge verifier failed; candidate was discarded | `integration_merge_id`, `candidate_merge_sha`, `discard_reason` (`"verifier_blocked"` \| `"crash_recovery"` \| `"merge_aborted"`) |
| `ReviewAggregationCompleted` | After every `SubmitReview` commit, when the cross-Reviewer logical-AND aggregator transitions out of `Pending` (the just-completed Reviewer was the LAST sibling to submit). Single-class observability event per [`audit-paired-writes.md ¬ß4`](audit-paired-writes.md); the underlying state mutation (`tasks.review_verdict` write + `Running ‚Üí Completed` transition) is already paired via `TaskStateChanged`. | `executor_task_id`, `triggered_by_reviewer_task_id`, `reviewer_count`, `verdict` ‚ąą `{"AllPassed", "AtLeastOneRejected", "NoSuccessors"}` |

All events are bound to the audit chain via the existing kernel
audit emission path; no new audit-side infrastructure is needed.

## ¬ß12 ‚ÄĒ Image Requirements (Verifier VMs)

Verifier VMs are operator-published per `INV-VM-CAP-03` with one
exception: the kernel-canonical `raxis-verifier-symbol-index`
image (per ¬ß14, `INV-VERIFIER-12`). Operators register every other
verifier image in `policy.toml [[vm_images]]` with `role_restriction`
including `Verifier`:

```toml
[[vm_images]]
name             = "raxis/parsers"
oci_digest       = "sha256:e3b0c44298fc1c149afbf4c8996fb924..."
role_restriction = ["Verifier"]                  # may include "Executor", "Orchestrator"

[[vm_images]]
name             = "raxis/rust-build"
oci_digest       = "sha256:c057a3e7ea75c2aef3c1cd95fa1aac84..."
role_restriction = ["Verifier", "Executor"]      # shared image; verifier-mode and executor-mode both use it
```

The `role_restriction` field is load-bearing: a plan declaring
`[[verifiers]].image = "raxis/parsers"` is admitted only if the
named image's `role_restriction` includes `Verifier`. This prevents
accidentally running a verifier on an image lacking the
`raxis-verifier` PID-1 binary.

### 12.1 Required Image Contents

Every verifier image (operator-published OR kernel-bundled) MUST
contain:

- `raxis-verifier` binary at `/usr/local/bin/raxis-verifier` (the
  kernel passes `/sbin/init ‚Üí raxis-verifier` by symlink at spawn).
  Statically linked recommended; dynamically linked acceptable
  with libc present.
- The trusted verifier wrapper binaries referenced by plan/policy
  `command` fields. Each `command` is launched directly as one
  absolute executable path; wrappers own their internal args/env.
- The toolchain the verifier command needs (`cargo`, `npm`,
  `pytest`, `ctags`, etc.) ‚ÄĒ operator's responsibility for
  operator-published images; kernel's responsibility for
  kernel-bundled starters per ¬ß14.
- Linux kernel ‚Č• 5.14 with cgroup v2 mounted; `cpu`, `memory`,
  `pids` controllers in `cgroup.subtree_control` ‚ÄĒ required by
  `INV-VERIFIER-06`.

Notably **may** contain (image-author choice):

- `git` ‚ÄĒ not strictly required (worktree is pre-populated by the
  kernel via VirtioFS), but useful if the verifier command needs
  `git log`, `git rev-parse`, etc.
- Network utilities ‚ÄĒ not by default; verifier VMs have no network
  interface unless operator declares an `allowed_egress` block.
- LSP servers ‚ÄĒ generally not needed (verifiers run commands, not
  semantic analysis); allowed if a verifier command genuinely
  depends.

The kernel does NOT inspect operator-published verifier image
contents for forbidden binaries (unlike the Reviewer image, where
it does ‚ÄĒ per [`planner-harness.md ¬ß10.4`](planner-harness.md)). Operator-published
verifier images are operator-trust domain. The kernel-canonical
`raxis-verifier-symbol-index` image is kernel-trust domain
(digest-verified at every spawn per ¬ß14.4).

## ¬ß13 ‚ÄĒ Invariants

> **`INV-VERIFIER-01` ‚ÄĒ Verifier output enters the audit chain via
> `witness_records` only.** A verifier VM cannot invoke
> `IntentKind::SingleCommit`, `CompleteTask`, `IntegrationMerge`, or
> any other commit-pathway intent. Verifier VMs have no planner
> harness, no LLM, no inference. Their only kernel-bound communication
> is the `WitnessSubmission` frame.

> **`INV-VERIFIER-02` ‚ÄĒ Verifier VMs are isolated from agent VMs.**
> No shared state; no inter-VM IPC. The only path from verifier
> output to a Reviewer is the kernel's KSB injection at Reviewer
> activation time (per ¬ß8). Verifier VMs cannot be observed by the
> Executor whose commit they evaluate, or by sibling Reviewers, or
> by the Orchestrator. Their existence is kernel-scope.

> **`INV-VERIFIER-03` ‚ÄĒ Reviewer activation is gated on completion
> of every plan-declared per-task verifier for the corresponding
> `evaluation_sha`.** A Reviewer is never activated with partial
> witness data. If any declared verifier is `Pending` (no witness
> yet), the Reviewer waits. If any declared verifier is
> `block_review`-failed, the Reviewer never activates and the task
> transitions to `Failed`.

> **`INV-VERIFIER-04` ‚ÄĒ `block_review` failures roll the originating
> Executor's `CompleteTask` into `Failed`** with the verifier output
> as the failure reason. The Executor receives `FAIL_VERIFIER_BLOCKED`
> on its next intent and may revise per [`agent-disagreement.md ¬ß2`](agent-disagreement.md).
> A `block_review` verifier failure counts as a review round for
> the purposes of `INV-CONVERGENCE-01`.

> **`INV-VERIFIER-05` ‚ÄĒ Declared artifacts MUST exist, be non-empty,
> and not exceed `artifact_max_bytes` post-success for the verifier
> to be considered passed.** Missing, empty, or oversize artifacts
> produce `final_status = "artifact_missing"` regardless of the
> command's exit code; the kernel does not partial-stage or
> truncate.

> **`INV-VERIFIER-06` ‚ÄĒ Verifier VM images MUST satisfy the same VM
> kernel and cgroup substrate requirements as planner images.**
> Linux 5.14+ guest kernel; cgroup v2 mounted; `cpu`, `memory`,
> `pids` controllers in `cgroup.subtree_control`. Verifier VMs use
> the same cgroup-based containment as planner VMs
> (`INV-PLANNER-HARNESS-03`). `raxis doctor`'s `vm-images` category
> enforces this for every operator-published verifier image
> referenced by an installed `policy.toml`.

> **`INV-VERIFIER-07` ‚ÄĒ Operator-published verifier VMs are
> policy-pinned by OCI digest** per `INV-VM-CAP-03`. The
> `role_restriction` field on `[[vm_images]]` MUST include
> `Verifier` for any image referenced by a per-task or pre-merge
> verifier declaration. **Exception:** the kernel-canonical
> `raxis-verifier-symbol-index` image (`INV-VERIFIER-12`) ‚ÄĒ that
> single image is kernel-bundled and digest-verified by the kernel
> rather than by operator policy.

> **`INV-VERIFIER-08` ‚ÄĒ Verifier VMs have NO LLM and NO planner
> harness.** Their PID 1 is `raxis-verifier`, a single-purpose
> command runner. No claw-code module is linked into
> `raxis-verifier`. No tool registry, no inference channel, no
> `IntentKind` dispatch. This is structural: a verifier VM that
> crashes due to a programming error in the verifier command cannot
> escalate the failure mode ‚ÄĒ it just produces a `crashed` witness.

> **`INV-VERIFIER-09` ‚ÄĒ Verifier VMs have read-write access to
> `/workspace`** (mounted from a fresh clone of `evaluation_sha` for
> per-task / claim-gate verifiers, or `candidate_merge_sha` for
> pre-merge verifiers) **and `/raxis/`** (for artifact output). All
> `/workspace` and `/raxis/` mutations are dropped at VM exit
> unless declared as artifacts per ¬ß6. Verifier VMs cannot persist
> mutations to the `main_repo` or any session-shared storage.

> **`INV-VERIFIER-10` ‚ÄĒ Verifier timeouts are kernel-enforced via
> `cgroup.kill` on the verifier-process cgroup at the declared
> `timeout`** (or the per-verifier kernel hard cap, whichever is
> smaller). Timeout produces `VerifierTimedOut` audit and treats
> the verifier as failed per its `on_failure` rule. The kernel
> does not rely on the verifier's internal timeout handling for
> this guarantee.

> **`INV-VERIFIER-11` ‚ÄĒ Verifier VMs have NO network interface by
> default.** Network egress requires explicit `allowed_egress`
> declaration in the corresponding plan-side block (per-task or
> pre-merge), mirroring the Executor / Orchestrator egress pattern.
> The default is air-gapped; verifiers that don't need network do
> not get one. (This also makes the supply-chain compromise of a
> verifier image less impactful: a compromised image cannot
> exfiltrate `evaluation_sha` or `candidate_merge_sha` contents
> without explicit egress declared in plan.)

> **`INV-VERIFIER-12` ‚ÄĒ The kernel-canonical
> `raxis-verifier-symbol-index` image is the sole exception to
> `INV-VM-CAP-03` for verifier images.** This image is bundled with
> the kernel distribution per [`system-requirements.md ¬ß8.1`](system-requirements.md),
> digest-verified at every spawn (parallel to the canonical
> Reviewer / Orchestrator image verification per
> `INV-PLANNER-HARNESS-02` / `INV-PLANNER-HARNESS-05`), and may NOT
> appear as an entry in operator `[[vm_images]]` (the alias
> `"raxis-verifier-symbol-index"` is reserved). Operators reference
> it via the reserved alias only; the kernel resolves it to the
> bundled image at activation time. Justification: symbol indexing
> is structural for the Pure-Static Reviewer per `planner-harness.md
> ¬ß4.1` (without it, the Reviewer is squinting at the diff
> unaided); making this one image kernel-canonical lets the
> ergonomics layer auto-inject a symbol-index verifier into every
> Executor task whose touched paths include source files
> ([`operator-ergonomics.md ¬ß4.2`](operator-ergonomics.md)), eliminating
> `WARN_REVIEWER_MISSING_SYMBOL_INDEX` as a default-state warning.
> Tiered language starters (Rust / Node / Python / Go per ¬ß14.5)
> are kernel-bundled but NOT kernel-canonical ‚ÄĒ they are
> operator-published-target-equivalent (registered in `[[vm_images]]`
> with their bundled digest).

> **`INV-VERIFIER-13` ‚ÄĒ Pre-merge verifier execution gates
> `IntegrationMerge` admission.** No `IntegrationMerge` advances
> main while any matching pre-merge verifier (per the
> `applies_to` filter in ¬ß16.3) has `final_status` other than
> `"passed"` for the candidate merged tree. The candidate merged
> tree is an orphan commit until verification completes; it is
> renamed-and-pushed only on full pass and discarded on any
> `block_merge` failure. Main is never partially advanced.

> **`INV-VERIFIER-14` ‚ÄĒ Symbol-index witness provenance is
> kernel-provisioned, never Executor-derived.** The
> `raxis-verifier-symbol-index` verifier (per `INV-VERIFIER-12`)
> is always activated against a kernel-provisioned clone of the
> evaluation tree ‚ÄĒ the kernel performs `git clone --shared
> --no-checkout main_repo.git ‚Üí /workspace`, then
> `git checkout <evaluation_sha>` inside the verifier VM, all
> over the kernel-mediated VirtioFS mount. The Executor's
> worktree is never the substrate. This closes a confused-deputy
> path: a malicious or buggy Executor cannot pre-stage source
> bytes that misrepresent the symbol topology
> (`evaluation_sha ‚Üí symbol_index` is a pure function of the
> commit object, not of any Executor-mutable state). The
> Reviewer's KSB `verifier_witnesses` section that surfaces the
> symbol index therefore has the same trust budget as the audit
> chain itself: bound by the kernel's content addressing, not
> by the Executor's behavior. Implementation note: the
> kernel-spawned helper that performs the provisioning is the
> `verifier_provision_workspace` step in `kernel/src/verifier/
> spawn.rs` and emits `VerifierWorkspaceProvisioned
> { evaluation_sha, clone_method: "kernel_share_no_checkout" }`
> exactly once before the verifier VM is started, so an
> auditor can verify provenance without instrumenting the
> verifier itself.

> **`INV-VERIFIER-15` ‚ÄĒ Verifier authenticated egress requires
> explicit per-image policy opt-in.** A verifier VM that declares
> `allowed_egress` defaults to **audit-only** mode: outbound
> requests are logged but unauthenticated, and credentials from
> `[[providers.credentials]]` / `[[permitted_credentials]]` are NOT
> injected by the egress proxy. Authenticated egress requires a
> matching `[[verifier_credentials.images]]` entry with
> `permit_authenticated = true` AND
> `[verifier_credentials].emergency_audit_only = false`. The
> per-image opt-in is the operator's deliberate decision point that
> a specific verifier image's supply chain is trusted enough to
> receive real credentials. The kill-switch
> `emergency_audit_only` lets operators flip every verifier back to
> audit-only during incident response without rewriting per-image
> rows. (See ¬ß16.7 for the full schema, resolution chain, and
> migration story from V2.0 default-authenticated.)

## ¬ß14 ‚ÄĒ Kernel-Bundled Verifier Images

Two classes of kernel-bundled verifier images ship with the kernel
distribution:

- **Kernel-canonical** (single image, structural): the
  `raxis-verifier-symbol-index` image per `INV-VERIFIER-12`.
- **Tiered language starters** (multiple images, defaulting target):
  Rust, Node, Python, Go starters per ¬ß14.5.

The two classes have different trust models and different
operator-facing surfaces.

### 14.1 The kernel-canonical symbol-index image

**Purpose.** Produces a symbol index artifact at
`/raxis/symbol_index.json` that the Reviewer reads via `read_file`
to resolve symbols across the diff without an LSP. Solves the
`WARN_REVIEWER_MISSING_SYMBOL_INDEX` problem structurally.

**Image contents (canonical manifest):**

| Component | Why |
|---|---|
| `raxis-verifier` (statically linked) | PID 1 |
| `/usr/local/bin/raxis-symbol-index` | Trusted wrapper launched directly by the verifier runner |
| `ctags` (universal-ctags, ‚Č• 6.0) | Symbol-extraction tool; produces JSON output |
| Minimal Linux 5.14+ guest kernel with cgroup v2 substrate per `INV-VERIFIER-06` | Required by all verifier images |

**Approximate size:** ~30 MB compressed. Boot time on KVM:
typically < 500 ms cold.

**Trust model:** kernel-canonical per `INV-VERIFIER-12`. Digest is
shipped in the kernel binary; verified at every spawn by the
kernel (not by operator policy). Tampering produces
`FAIL_CANONICAL_VERIFIER_IMAGE_DIGEST_MISMATCH` and a
`SecurityViolationDetected` audit event (parallel to
`FAIL_REVIEWER_IMAGE_DIGEST_MISMATCH` per `policy-plan-authority.md
¬ß3b`).

**Standard verifier declaration (auto-injected by `plan prepare`
per [`operator-ergonomics.md ¬ß4.2`](operator-ergonomics.md) when policy `[prepare]
auto_inject_symbol_index = true`):**

```toml
[[plan.tasks.<id>.verifiers]]
name        = "symbol_index"
image       = "raxis-verifier-symbol-index"           # the reserved canonical alias
command     = "/usr/local/bin/raxis-symbol-index"
timeout     = "60s"
on_failure  = "warn_only"
artifact    = "/raxis/symbol_index.json"
artifact_max_bytes = 4194304                          # 4 MiB
```

The `command` and `artifact_max_bytes` are policy-defaultable per
[`operator-ergonomics.md ¬ß4.2`](operator-ergonomics.md) so deployments can tune (e.g., point
at `lib/` instead of `src/` for non-standard layouts) without
disabling the auto-inject.

### 14.2 Why `raxis-verifier-symbol-index` is canonical (not just bundled)

Three reasons distinguish it from the tiered starters:

1. **The Reviewer's KSB rendering depends on the artifact format.**
   The KSB has a `verifier_witnesses` section that calls out
   `symbol_index` artifacts specifically (per ¬ß8). The artifact
   schema (a particular ctags JSON dialect) is part of the
   Reviewer's input contract; if operators publish their own
   symbol-index image with a different schema, the Reviewer's
   training won't recognize it.
2. **Auto-injection requires a stable image alias.** The
   ergonomics layer ([`operator-ergonomics.md ¬ß4.2`](operator-ergonomics.md)) auto-injects
   a `symbol_index` verifier; that auto-inject must reference an
   image alias that's guaranteed to resolve. Kernel-canonical
   handles this without requiring operator setup.
3. **Trust surface is microscopic.** ctags has no diff-derived
   code execution paths, no postinstall hooks, no transitive
   supply chain (it's a standalone C program). The kernel team
   can keep the image current without significant ongoing CVE
   work.

### 14.3 The reserved alias `"raxis-verifier-symbol-index"`

Operators MAY NOT declare a `[[vm_images]]` entry with `name =
"raxis-verifier-symbol-index"`. Doing so produces
`FAIL_POLICY_RESERVED_VM_IMAGE_NAME` at policy load. The alias is
reserved so that plan-side references resolve unambiguously to the
kernel-bundled image.

Plan-side references like `image = "raxis-verifier-symbol-index"`
in `[[verifiers]]` blocks resolve via a special resolution path:
the kernel checks the reserved-aliases table first; if the alias
matches, it uses the kernel-bundled image; otherwise it falls
through to the standard `[[vm_images]]` lookup.

### 14.4 Digest verification at spawn

At every verifier-VM activation that uses the
`raxis-verifier-symbol-index` alias, the kernel:

1. Looks up the bundled image's expected SHA-256 from the
   kernel-binary-embedded canonical-images manifest.
2. Reads the on-disk image at `<data_dir>/canonical_images/raxis-verifier-symbol-index-<kernel_version>.img`.
3. Computes the actual SHA-256.
4. Compares; on mismatch:
   - Aborts the spawn before VM boot.
   - Emits `SecurityViolationDetected { kind: "canonical_verifier_image_digest_mismatch", expected, actual }`.
   - Returns `FAIL_CANONICAL_VERIFIER_IMAGE_DIGEST_MISMATCH` to the
     verifier-spawn caller.
5. The kernel halts further verifier spawns of any kind (defense
   in depth ‚ÄĒ if one canonical image was tampered, others may be
   too) until `raxis doctor canonical-images` succeeds.

This mirrors the Reviewer / Orchestrator canonical image
verification semantics from `policy-plan-authority.md ¬ß3b
FAIL_REVIEWER_IMAGE_DIGEST_MISMATCH` and
`FAIL_ORCHESTRATOR_IMAGE_DIGEST_MISMATCH`.

### 14.5 Tiered language starters (Rust / Node / Python / Go)

The kernel distribution bundles four general-purpose verifier
starter images for the most common language ecosystems:

| Starter alias | Approximate size | Bundled toolchain |
|---|---|---|
| `raxis-verifier-rust-starter` | ~600 MB | `cargo`, `rustc`, `clippy`, `rustfmt`, `cargo-nextest` (latest stable Rust at kernel release time) |
| `raxis-verifier-node-starter` | ~400 MB | `node` (LTS), `npm`, `pnpm`, `eslint`, `tsc` |
| `raxis-verifier-python-starter` | ~300 MB | `python3` (latest stable), `pip`, `pytest`, `mypy`, `ruff` |
| `raxis-verifier-go-starter` | ~250 MB | `go` (latest stable), `gofmt`, `staticcheck`, `golangci-lint` |

**Trust model.** Tiered starters are **kernel-bundled but NOT
kernel-canonical**. The distinction:

- They ship with the kernel distribution (`system-requirements.md
  ¬ß8.1`).
- The kernel team is responsible for keeping them current and
  CVE-patched at each kernel release.
- Operators register them in `policy.toml [[vm_images]]` with the
  bundled digest, exactly as if they were operator-published. The
  kernel does NOT auto-resolve their aliases (unlike
  `raxis-verifier-symbol-index`).
- Operators may pin to the bundled digest, override with their own
  fork's digest, or simply omit the starter entirely.

This trust model lets operators audit the bundled images via the
same path as any other image in `[[vm_images]]`, and lets them
opt out of any starter without affecting kernel structural
guarantees.

**Why tiered, not one mega-image.** Single-toolchain starters keep
each image small (sub-second boot times), keep the trust surface
per-starter narrow, and let operators register only the languages
they actually use. A mega-image with all four toolchains would be
~1.5 GB and would force every operator to inherit CVE exposure
for languages they don't use.

**Setup wizard integration.** [`operator-ergonomics.md ¬ß16.3`](operator-ergonomics.md) adds a
"language stacks" phase that scans the main_repo for language
hints (`Cargo.toml`, `package.json`, `pyproject.toml`, `go.mod`)
and pre-checks the corresponding starters for registration. The
wizard writes the appropriate `[[vm_images]]` entries with the
kernel-bundled digests.

**`[default_verifier_images]` policy schema** (per
[`policy-plan-authority.md ¬ß4`](policy-plan-authority.md)):

```toml
[default_verifier_images]
rust   = "raxis-verifier-rust-starter"
node   = "raxis-verifier-node-starter"
python = "raxis-verifier-python-starter"
go     = "raxis-verifier-go-starter"
```

The mapping is consumed by `plan prepare` for plan-author
ergonomics: a verifier declaration with `image = "@rust"` (or
`@node`, `@python`, `@go`) resolves through this table at
prepare-time, getting filled with the operator's preferred image
alias and an `# @raxis-default v0.4.0` annotation. Plan authors
who want a specific image continue to write the alias verbatim;
the `@<lang>` shortcut is a convenience for the common case.

### 14.6 Doctor integration

`raxis doctor canonical-images` (per [`system-requirements.md ¬ß11`](system-requirements.md))
verifies the symbol-index image's digest specifically (treating
mismatch as fatal per ¬ß14.4). A new
`raxis doctor verifier-starters` category verifies the four
tiered starters' digests informationally ‚ÄĒ mismatch produces a
warning, not a fatal error, since operators may have intentionally
deviated.

## ¬ß15 ‚ÄĒ Pre-IntegrationMerge Verifier Plan Declaration

Pre-merge verifiers run when the Orchestrator submits
`IntegrationMerge`. They evaluate the **candidate merged tree**
(an orphan commit produced by [`integration-merge.md ¬ß4 Check 5d`](integration-merge.md)
that is discarded on `block_merge` failure and renamed-to-main on
pass). They are the channel for catching cross-task interactions
and validating the actually-merged state.

### 15.1 Plan-author surface

```toml
[[plan.integration_merge_verifiers]]
name        = "e2e_smoke"
image       = "raxis-verifier-node-starter"
command     = "/usr/local/bin/raxis-node-e2e-smoke"
timeout     = "30m"
on_failure  = "block_merge"                      # block_merge | warn_only
applies_to  = "all"                              # all | task_set | last; defaults to "all"

# task_set option:
[[plan.integration_merge_verifiers]]
name        = "auth_integration"
image       = "raxis-verifier-rust-starter"
command     = "/usr/local/bin/raxis-auth-integration"
timeout     = "20m"
on_failure  = "block_merge"
applies_to  = "task_set"
task_set    = ["implement_auth", "implement_session"]   # only when these tasks contributed to this merge

# last option:
[[plan.integration_merge_verifiers]]
name        = "deploy_smoke"
image       = "operator/deploy-smoke@sha256:..."
command     = "./scripts/deploy_smoke.sh"
timeout     = "10m"
on_failure  = "block_merge"
applies_to  = "last"                             # only the FINAL IntegrationMerge of the DAG
```

### 15.2 Operator-global surface

Operators may declare deployment-wide pre-merge gates in
`policy.toml`:

```toml
# policy.toml
[[integration_merge_verifiers]]
name                      = "production_deploy_smoke"
image                     = "operator/deploy-smoke@sha256:..."
command                   = "./scripts/prod_smoke.sh"
timeout                   = "20m"
on_failure                = "block_merge"
required_for_environments = ["production"]       # composes with environment binding (environment-access-control.md)
```

Operator-side declarations:

- Cannot be downgraded to `warn_only` by any plan.
- Compose with environment binding via `required_for_environments`
  ‚ÄĒ a verifier in this list runs only when the candidate merged
  tree's `IntegrationMerge` resolves (per
  `environment-access-control.md INV-ENV-01`) into one of the
  named environments. Empty or omitted `required_for_environments`
  means "always runs."
- Use the same `applies_to` field (defaulting to `"all"`); operators
  with multi-merge DAGs can scope their global gates similarly to
  plan-author ones.

### 15.3 Field semantics (pre-merge)

| Field | Required? | Plan-author | Operator | Notes |
|---|---|---|---|---|
| `name` | Required | Yes | Yes | `[a-z][a-z0-9_]{0,31}`; unique within source |
| `image` | Required | Yes | Yes | Resolves to `[[vm_images]]` with `Verifier` role_restriction |
| `command` | Required | Yes | Yes | Single absolute executable path launched directly; cwd `/workspace` (mounted from candidate merged tree); args/env belong inside the wrapper |
| `timeout` | Required | Yes | Yes | ‚Č• 5s, ‚Č§ kernel hard cap |
| `on_failure` | Required | `block_merge` \| `warn_only` | `block_merge` only (cannot downgrade to `warn_only`) | `block_review` is rejected (`FAIL_VERIFIER_INVALID_ON_FAILURE`) |
| `applies_to` | Optional | `all` \| `task_set` \| `last` (default `all`) | Same | See ¬ß16.3 |

## ¬ß16 ‚ÄĒ Pre-Merge Verifier Lifecycle Detail

### 16.1 Hook trigger point

Triggered at [`integration-merge.md ¬ß4 Check 5d`](integration-merge.md), immediately after
Check 5c (universal merge approval gate). At this point the
Orchestrator has already passed:

- Check 1 (Dispatch matrix ‚ÄĒ Orchestrator-only)
- Check 2 (commit_sha reachability)
- Check 3 (Ancestry verification)
- Check 4 (merged_task_ids validation ‚ÄĒ all tasks Completed)
- Check 5 (Diff computation + hybrid allowlist)
- Check 5b (Protected path approval gate)
- Check 5c (Universal merge approval gate)

So pre-merge verifiers fire only on merges that have already passed
all the cheaper structural and operator-approval gates. The cost
discipline is that pre-merge verifiers don't waste cycles on
malformed merges.

### 16.2 Candidate merged tree creation

Per [`integration-merge.md ¬ß4 Check 5d.2`](integration-merge.md):

1. Kernel computes the merge commit that *would* result from
   `IntegrationMerge { commit_sha, merged_task_ids }` ‚ÄĒ using the
   same git operations the Orchestrator's `git merge` invocation
   would use, but produced as an **orphan commit** in the kernel's
   verifier-staging area:
   ```text
   $RAXIS_DATA_DIR/candidate_merges/<integration_merge_id>/
   ```
   This is a separate worktree, NOT on main, NOT on any task
   branch.
   In the V2.7 `{ base_sha, head_sha }` wire shape, the kernel uses
   the mechanically derived "all completed Executor tasks" set from
   [`integration-merge.md ¬ß4 Check 4`](integration-merge.md) as the
   effective `merged_task_ids`.
2. Kernel records `candidate_merge_sha` in the
   `integration_merge_attempts` table (new column) for crash
   recovery.
3. Emits `CandidateMergeTreeCreated { integration_merge_id,
   candidate_merge_sha, merged_task_ids }`.

If the candidate merge fails to compute (e.g., the Orchestrator
submitted a malformed `commit_sha`), Check 5d returns
`FAIL_CANDIDATE_MERGE_COMPUTATION_FAILED` and no verifiers spawn.
Discard logic per ¬ß16.5.

### 16.3 `applies_to` resolution

For each `[[plan.integration_merge_verifiers]]` and
`[[integration_merge_verifiers]]` declared (the union of plan-author
and operator-global):

```rust
fn matches_current_merge(verifier, merge) -> bool {
    match verifier.applies_to {
        "all" => true,
        "task_set" => {
            // Intersect declared task_set with the current merge's merged_task_ids.
            verifier.task_set.iter().any(|t| merge.merged_task_ids.contains(t))
        }
        "last" => {
            // True if and only if this is the final merge of the DAG.
            // Computed: no remaining tasks reachable in the DAG that are not
            // in this merge's merged_task_ids and not yet merged in a prior
            // IntegrationMerge.
            is_final_merge_of_dag(merge, plan.dag)
        }
    }
    && environment_filter_matches(verifier, merge)   // operator-only required_for_environments
}
```

**`applies_to = "last"` semantics for parallel branches.** The
"final merge" is defined as: no task in the DAG remains in
`Pending` / `Active` state AND no task in the DAG remains in
`Completed` state without having been merged. This handles parallel
branches correctly ‚ÄĒ the last `IntegrationMerge` is whichever one
clears the final remaining `Completed` task. Operators with complex
multi-branch DAGs should test their `applies_to = "last"` verifiers
on representative DAG shapes; the kernel's
`raxis-cli plan explain --merge-coverage` command renders which
merges each verifier will fire on.

### 16.4 Gating and main advance

Per ¬ß5.3 algorithm:

- All matching verifiers complete with `final_status` in {passed,
  failed, timed_out, crashed, artifact_missing}.
- If any `block_merge` verifier is non-passing: discard candidate
  per ¬ß16.5; return `FAIL_INTEGRATION_MERGE_VERIFIER_BLOCKED` to
  Orchestrator; NO main advance.
- If all `block_merge` verifiers pass: candidate is renamed-to-main
  per the standard `IntegrationMerge` advance flow (Check 6 onward
  in [`integration-merge.md ¬ß4`](integration-merge.md)).

### 16.5 Discard on failure

When pre-merge verification fails or the merge is otherwise
aborted:

1. Kernel removes the candidate worktree at
   `candidate_merges/<integration_merge_id>/`.
2. Kernel marks `integration_merge_attempts` row with
   `discard_reason ‚ąą {"verifier_blocked", "crash_recovery",
   "merge_aborted"}`.
3. Kernel runs `git gc --prune=now` against `main_repo` (the
   parent of the orphan candidate commit) under a short-timeout
   advisory lock ‚ÄĒ see [`integration-merge.md ¬ß11.10.3`](integration-merge.md)'s
   `git_gc_orphan` helper. Lock contention with an in-flight
   IntegrationMerge phase 2 is a recognized skip case; the orphan
   is then collected by the periodic `git_maintenance_main` job.
4. Kernel emits `CandidateMergeTreeDiscarded { integration_merge_id,
   candidate_merge_sha, discard_reason }`.

The retention story for the orphan commit is the union of the
synchronous step 3 above and the periodic
`git_maintenance_main` job
([`kernel-lifecycle.md ¬ß10.5.3`](kernel-lifecycle.md), cadence 6 h with opportunistic
disk-pressure trigger). With both layers in place, a high
failure-rate burst of pre-merge verifications can no longer fill the
disk between cadence ticks: each failed attempt reaps its own orphan
in step 3, and the periodic sweep is a backstop for cases where the
synchronous reap could not acquire the repo lock. Main is never
polluted.

### 16.6 Orchestrator failure handling

When the Orchestrator receives `FAIL_INTEGRATION_MERGE_VERIFIER_BLOCKED`:

1. Per [`agent-disagreement.md ¬ß6`](agent-disagreement.md), the Orchestrator typically
   cannot self-resolve a pre-merge regression (the contributing
   tasks already passed their per-task verifiers and per-task
   Reviews). The Orchestrator's NNSP routes this as an operator
   escalation: `EscalationRequest { class:
   IntegrationMergeRegression, ... }`.
2. The escalation includes the `candidate_merge_sha`,
   `verifier_names`, and `primary_witness_summary`.
3. The operator decides:
   - Approve a workaround (e.g., a follow-up plan that fixes the
     integration regression).
   - Mark the contributing tasks as `Failed` for re-work via a
     subsequent initiative.
   - Abandon the initiative.
4. The candidate merged tree is permanently discarded; if the
   operator wants to re-attempt the same merge composition after
   fixing things, the Orchestrator submits a fresh
   `IntegrationMerge` and the kernel produces a new candidate
   tree.

Pre-merge verifier failures do NOT count toward
`INV-CONVERGENCE-01` because they are not review rounds ‚ÄĒ they
are post-Reviewer regressions that require operator judgment
rather than agent retry.

## ¬ß16.7 ‚ÄĒ Verifier Credential Injection (audit-only by default)

> **Status:** V2 normative.
> **Cross-references:** `INV-VERIFIER-11` (default no network),
> [`credential-proxy.md`](credential-proxy.md) (credential injection mechanism for non-verifier
> roles), [`key-revocation.md ¬ß6`](key-revocation.md) (provider-credential lifecycle that
> applies to verifier credentials too).

### ¬ß16.7.1 ‚ÄĒ Why a separate credential model for verifiers

Verifier images are the **weakest link in the supply chain**. Unlike
Executor images (kernel-provisioned from a known base image) and
Reviewer / Orchestrator images (kernel-canonical, digest-bound per
`INV-PLANNER-HARNESS-02` / `INV-PLANNER-HARNESS-05`), verifier images
may be:

- Operator-authored against a starter (`raxis-verifier-rust-starter`,
  etc., kernel-bundled but operator-extended).
- Third-party-authored ‚ÄĒ e.g., a security firm publishes
  `corp-verifier-sca:v3` for software-composition-analysis verification.
- Community-maintained ‚ÄĒ e.g., a public registry `community-verifier-*`
  series for common toolchains.

The blast radius of a compromised verifier with authenticated egress
is wider than a compromised Executor with authenticated egress:

- The Executor touches **only its own task's worktree**; a leaked
  Executor egress credential can exfiltrate that task's code (already
  scoped to the task's path-allowlist) and consume that task's budget.
- A verifier touches **every task's code** ‚ÄĒ it runs on
  `evaluation_sha`, which is the candidate-merge tree of all
  contributing tasks. A leaked verifier egress credential can
  exfiltrate the entire integration delta. If the credential is
  authenticated against a private registry, the verifier can
  **publish** to that registry ‚ÄĒ every downstream consumer of that
  registry is then compromised.

V2 therefore makes the **default safe**: verifier VMs that declare
`allowed_egress` get a proxy that LOGS every outbound request with
full URL / headers / body-size telemetry, but NEVER injects real
credentials. The verifier sees `HTTP 401 Unauthorized` from the
upstream and logs it; the operator sees `VerifierEgressAttempted`
audit events with the verifier image, the URL, and the unauthenticated
status. This lets operators observe what a verifier *would* fetch
without ever exposing real credentials to a potentially-compromised
verifier image.

For verifiers that genuinely need authenticated egress (e.g., a
verifier that downloads a private NPM dependency manifest, or a
verifier that hits a private security-scanning API), the operator
**explicitly opts in** with a per-image `permit_authenticated` flag
in `policy.toml`. This creates a deliberate decision point: the
operator is saying "I trust this specific verifier image enough to
give it real credentials."

### ¬ß16.7.2 ‚ÄĒ `policy.toml [verifier_credentials]` schema

```toml
# policy.toml
#
# Top-level [verifier_credentials] = global defaults / kill-switch.
# [[verifier_credentials.images]] = per-image opt-in to authenticated
#                                   egress.
#
# Plan-side: verifiers declare allowed_egress (per existing schema).
# What this section controls is whether that egress is authenticated
# or audit-only.

[verifier_credentials]
# Global default. true  ‚Üí authenticated by default for any verifier
# image that has allowed_egress declared (legacy behavior; V2.0 implicit
# default; not recommended for V2.1+).
# false ‚Üí audit-only by default; per-image permit_authenticated must be
# set to true to enable real credential injection. (Recommended; V2.1
# default.)
default_authenticated = false

# Optional global kill-switch. If true, NO verifier image gets
# authenticated egress regardless of per-image permit_authenticated.
# Operators flip this during incident response when supply-chain
# integrity is in question.
emergency_audit_only = false

[[verifier_credentials.images]]
# Per-image opt-in. Image MUST match a [[vm_images]] entry's `name`
# (verifier-processes.md ¬ß12). The kernel rejects with
# FAIL_VERIFIER_CREDENTIAL_UNKNOWN_IMAGE if image is not declared.
image                  = "raxis-verifier-rust-starter:v2"
permit_authenticated   = true
# Optional: scope which credentials this image may receive. If absent,
# the image may receive any credential listed in its declared
# allowed_egress. Names match [[providers.credentials]].name and
# [[permitted_credentials]].name.
credentials            = ["npm-registry-token", "github-pat"]
# Optional: explicit policy reference for the operator's audit trail
# justifying the opt-in. Forensic-only.
authorized_by          = "secops-2026-q2"
authorization_reference = "TICKET-SEC-1729"

[[verifier_credentials.images]]
image                  = "community-verifier-python-linter:latest"
permit_authenticated   = false   # default ‚ÄĒ explicit for clarity
```

### ¬ß16.7.3 ‚ÄĒ Effect on the `allowed_egress` resolution chain

When a verifier VM is spawned with `allowed_egress` declarations,
the kernel walks the ¬ß16.7.2 schema as follows (inside the same
`approve_plan` shift-left chain that processes other plan validation):

1. **No `allowed_egress` declared.** The VM has no network interface
   per `INV-VERIFIER-11`; nothing to do; spawn proceeds.
2. **`allowed_egress` declared.** The kernel determines the image's
   credential mode:
   - If `[verifier_credentials].emergency_audit_only == true` ‚Üí
     **audit-only** (override).
   - Else if a `[[verifier_credentials.images]]` row matches the
     image AND `permit_authenticated == true` ‚Üí **authenticated**
     (subject to the row's `credentials` filter, if present).
   - Else if `[verifier_credentials].default_authenticated == true`
     ‚Üí **authenticated** (legacy V2.0 behavior; emits a
     `WARN_VERIFIER_AUTHENTICATED_BY_GLOBAL_DEFAULT` at admission so
     operators see the implicit grant).
   - Else ‚Üí **audit-only** (V2.1 default).
3. The kernel sets the verifier VM's egress proxy mode accordingly.
   In **audit-only** mode, the proxy:
   - Strips any `Authorization`, `Cookie`, and operator-known
     credential headers from outbound requests (defense-in-depth: even
     if the verifier image's environment contains a stale token, it's
     not forwarded).
   - Does not inject credentials from `[[providers.credentials]]` or
     `[[permitted_credentials]]`.
   - Logs every outbound request as
     `VerifierEgressAttempted { verifier_run_id, image, method,
     scheme, host, path, query_keys, header_count, body_size_bytes,
     unauthenticated_response_code }`.
   - Forwards the request to the upstream WITHOUT authentication.
     The upstream typically returns `401 Unauthorized` or `403
     Forbidden`; the verifier observes this and decides how to
     handle (typically: emit a `failed` witness with a clear error
     message).
4. In **authenticated** mode, the proxy follows the existing
   [`credential-proxy.md`](credential-proxy.md) injection model, scoped to the
   credentials listed in the image's row (if any).

### ¬ß16.7.4 ‚ÄĒ Audit events

```rust

    verifier_run_id:    Uuid,
    task_id:            Option<TaskId>,        // None for pre-merge
    integration_merge_id: Option<Uuid>,        // Some for pre-merge
    image:              String,
    mode:               VerifierCredentialMode, // AuditOnly | Authenticated
    resolution_path:    String, // e.g., "emergency_audit_only" |
                                //       "image_permit_authenticated" |
                                //       "global_default_authenticated" |
                                //       "global_default_audit_only"
    emitted_at_ms:      u64,
}

AuditEventKind::VerifierEgressAttempted {
    verifier_run_id:    Uuid,
    image:              String,
    method:             String,
    scheme:             String,
    host:               String,
    path:               String,
    query_keys:         Vec<String>,        // sorted; values redacted
```

### ¬ß16.7.5 ‚ÄĒ Failure codes

| Code | Trigger |
|---|---|
| `FAIL_VERIFIER_CREDENTIAL_UNKNOWN_IMAGE` | A `[[verifier_credentials.images]]` row references an image not declared in `[[vm_images]]`. Caught at policy load. |
| `FAIL_VERIFIER_CREDENTIAL_UNKNOWN_CREDENTIAL` | A `[[verifier_credentials.images]].credentials` entry names a credential not declared in `[[providers.credentials]]` or `[[permitted_credentials]]`. Caught at policy load. |
| `FAIL_VERIFIER_AUTHENTICATED_EGRESS_DENIED_BY_POLICY` | Plan declares `allowed_egress` AND policy resolution lands in `audit-only` AND the verifier (at runtime) attempts to use a credential that the operator's plan-side declaration assumed available. Surfaced as a verifier `crashed` witness with this reason in `error_text`. |

### ¬ß16.7.6 ‚ÄĒ Migration from V2.0

V2.0 deployments that did not have `[verifier_credentials]` declared
behave identically to V2.0 (authenticated-by-default), because
the absence of the section leaves `default_authenticated` unset; a
forward-compatible parser treats that case as `true` for V2.0
deployments and emits a one-shot
`WARN_V2_0_VERIFIER_CREDENTIALS_DEFAULT_DEPRECATED` audit warning.
V2.1 deployments that explicitly want the V2.0 behavior must opt in
via `default_authenticated = true`. Operators upgrading to V2.1 MUST
audit their verifier images and either:

- Add `[[verifier_credentials.images]]` rows for the verifiers that
  legitimately need authenticated egress, OR
- Accept the V2.1 audit-only default and tolerate the `401`s in
  verifier output (which usually surface as `failed` witnesses with
  clear error messages, prompting the operator to add the
  per-image opt-in).

The migration bias is intentional: the operator must take a
deliberate action to grant authenticated egress to a verifier image,
making the decision visible in the operator's policy diff and the
audit chain.

### ¬ß16.7.7 ‚ÄĒ Composition with `INV-VERIFIER-11`

`INV-VERIFIER-11` ("verifier VMs have no network by default") is
unchanged: a verifier with NO `allowed_egress` declaration gets
NO network interface, regardless of `[verifier_credentials]`. The
new spec only governs what happens when `allowed_egress` IS
declared:

| `allowed_egress` declared? | `permit_authenticated`? | `emergency_audit_only`? | Outcome |
|---|---|---|---|
| No | (n/a) | (n/a) | No network interface (`INV-VERIFIER-11`). |
| Yes | Yes | No | Authenticated egress (existing V2.0 behavior). |
| Yes | No (or row absent) | No | Audit-only egress (V2.1 default). |
| Yes | (any) | Yes | Audit-only egress (kill-switch override). |

**"Audit-only"** means the request is forwarded to the upstream without credential injection; the verifier receives the upstream's unauthenticated response (typically `HTTP 401`). Public endpoints that require no authentication will succeed normally. Every request is logged as a `VerifierEgressAttempted` audit event regardless of outcome.

## ¬ß18 ‚ÄĒ Out of V2 Scope

- **Per-`SingleCommit` verifier execution.** Per-commit verification
  was evaluated and rejected ‚ÄĒ the cost explosion (N commits √ó
  M verifiers per task) doesn't pay for the marginal feedback
  benefit, and the agent's own `bash` tool is the right place for
  tight per-commit feedback loops.
- **Periodic / time-based verifier execution.** Background
  processes the Executor itself manages (per `planner-harness.md
  ¬ß5`) cover the legitimate "is the dev server still up?" use
  cases. Kernel-managed periodic verifiers add stateful complexity
  for negligible benefit.
- **Manual / on-demand operator-triggered verifier runs.** Possible
  V3 debug feature; not core flexibility.
- **Cross-task artifact sharing** beyond the per-Reviewer staging
  scoping in ¬ß6.4. Future enhancement if a real need emerges.
- **Verifier output streaming to consumers mid-execution.**
  Consumers see complete witness data only; partial results would
  complicate the gating algorithms in ¬ß5.2 / ¬ß5.3 with ambiguity
  about partially-complete verifiers.
- **Operator-defined `final_status` values.** The five values
  (`passed`, `failed`, `timed_out`, `crashed`, `artifact_missing`)
  are exhaustive at the kernel layer. Verifiers can encode finer
  status in `structured_counters` if they want; the kernel's
  routing is binary (passed vs. not-passed).
- **Reviewer-side or Orchestrator-side verifier re-execution
  requests.** Neither role can re-run a verifier. If they believe
  a verifier's result is wrong (e.g., a flaky test), they must
  escalate per [`agent-disagreement.md ¬ß6`](agent-disagreement.md).
- **Inter-verifier dependencies within a hook.** Verifiers run in
  parallel within a hook invocation; sequential dependencies are
  encoded inside one trusted check wrapper executable.
- **V3 audit retention treatment of artifact bytes.** Audit events
  reference artifacts by `artifact_sha256` and `artifact_size_bytes`;
  the bytes themselves live in the per-session staging directory
  (cleaned at task termination). V3 audit retention may add
  long-term retention of artifact bytes for forensic replay.

## ¬ß20 Operator-authored hints + bounded-runtime guard

This section closes two previously-queued follow-up items:

### ¬ß20.1 Operator hints

Operators may declare a `hints: BTreeMap<String, serde_json::Value>` map
on every verifier surface:

* `[[gates]] hints` (claim-based)
* `[[integration_merge_verifiers]] hints` (operator-side, V2)
* `[[plan.integration_merge_verifiers]] hints` (plan-side, V2)

The hints are validated at policy/plan load time
(`validate_verifier_hints` in `crates/policy/src/bundle.rs`,
shared between operator-side and plan-side validators) against the
following discipline:

* ‚Č§ 32 entries
* ‚Č§ 4 KiB total when serialised via `serde_json::to_vec`
* No key may start with `RAXIS_` (the reserved kernel-injected env
  prefix; operator-spoofing of kernel envs is rejected at load).

### ¬ß20.1.1 Spawn-envelope echo

The kernel injects the deterministic JSON serialisation of the
hints map into the verifier spawn envelope as a single env var:

```text
RAXIS_VERIFIER_OPERATOR_HINTS_JSON={"language":"rust","coverage_min_pct":85}
```

The env var is ALWAYS set (empty `{}` when no hints declared), so a
verifier may read it unconditionally. The `BTreeMap` ordering on the
kernel side guarantees a byte-stable wire shape ‚ÄĒ the verifier can
hash the env var bytes directly if it needs cache-key shape.

### ¬ß20.1.2 Witness-body echo

After token validation + evaluation-SHA binding, BEFORE persisting
the witness, the kernel:

1. Looks up the gate's policy-declared hints by `gate_type`.
2. If the verifier's claimed `body` is a JSON object containing the
   reserved key `operator_hints`, REJECTS the submission with
   `WitnessRejectionReason::SpoofedOperatorHints` and emits
   `AuditEventKind::WitnessOperatorHintSpoofingDetected`. Token is
   NOT consumed.
3. Otherwise, when policy hints are non-empty AND the body is a JSON
   object, inserts the policy-declared map under the reserved key.
4. Hashes the (now-augmented) body bytes for blob storage.

The kernel is the only source of truth for `body.operator_hints` ‚ÄĒ
the verifier is never trusted to echo hints back faithfully.
Pinned by `INV-WITNESS-OPERATOR-HINTS-ECHOED-01` and
`INV-WITNESS-OPERATOR-HINT-SPOOFING-REJECTED-01`.

Deferred: per-gate-type rich hint schema layer; operator-hint
surfacing for non-Object witness bodies.

### ¬ß20.2 Bounded-runtime guard

The new `[verifier_runtime]` policy section ships four operator-
tunable fields with the following spec defaults:

| Field                                       | Default | Hard cap   |
|---------------------------------------------|---------|------------|
| `max_verifier_wall_seconds`                 | 300     | 4 hours    |
| `verifier_idle_timeout_seconds`             | 60      | 30 minutes |
| `task_verifier_total_budget_seconds`        | 900     | 8 hours    |
| `verifier_force_shutdown_grace_seconds`     | 10      | 5 minutes  |

Coherence rule: `verifier_idle_timeout_seconds ‚Č§ max_verifier_wall_seconds`.

Six guards land alongside the new fields:

1. **Hard wall-clock kill** in `gates::verifier_runner::spawn_verifier_with_audit`
   ‚ÄĒ `tokio::time::timeout(min(declared_timeout, policy_max), child.wait())`.
   On expiry: SIGTERM via `start_kill()`, wait `verifier_force_shutdown_grace_seconds`,
   SIGKILL via `kill()`. Emits `VerifierWallClockTimeout`. INV-VERIFIER-WALL-CLOCK-KILL-01.
2. **Idle-stream timeout** ‚ÄĒ every UDS read on `RAXIS_KERNEL_SOCKET`
   is wrapped in `tokio::time::timeout(verifier_idle_timeout, ...)`.
   On expiry: same reap path; emits `VerifierIdleTimeout`. INV-VERIFIER-IDLE-TIMEOUT-01.
3. **Per-task cumulative-time ceiling** ‚ÄĒ in-memory
   `OnceLock<Mutex<BTreeMap<TaskId, u64>>>` tracker accumulates wall
   time across retries. When the ceiling is crossed, the next spawn
   returns `GateError::VerifierBudgetExhausted` + emits
   `VerifierBudgetExhausted`. INV-VERIFIER-CUMULATIVE-BUDGET-01.
4. **VM force-shutdown grace** ‚ÄĒ new `Session::shutdown_grace_then_force(grace)`
   trait method (default impl delegates to `shutdown(grace)`). Apple-VZ
   override issues an explicit graceful-then-force dance. On escalation,
   returns `ExitStatus::BackendError`; kernel emits
   `VerifierVmForcedShutdown`. INV-VERIFIER-VM-FORCE-SHUTDOWN-01.
5. **Witness-handler bounded time** ‚ÄĒ `handlers::witness::handle`
   wraps its body in `tokio::time::timeout(5s, ...)`. On expiry:
   emits `WitnessHandlerTimeout` + returns `HandlerError::HandlerTimedOut`.
   INV-WITNESS-HANDLER-BOUNDED-01.
6. **Audit emission** ‚ÄĒ every kill path emits exactly one of the
   five audit variants (`VerifierWallClockTimeout`,
   `VerifierIdleTimeout`, `VerifierBudgetExhausted`,
   `VerifierVmForcedShutdown`, `WitnessHandlerTimeout`) +
   `WitnessOperatorHintSpoofingDetected` for the operator-hint
   spoofing case. All six variants are wired through the SSE bridge
   in `kernel/src/notifications/sink.rs` so the dashboard observes
   them in real time.
