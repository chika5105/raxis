================================================================================
  RAXIS E2E Realistic-Scenario Dry-Run — Every Code Path, Step by Step
================================================================================

  Purpose
  -------
  This is a forensic dry-run of `extended_e2e_realistic_scenario.rs::
  realistic_session_lifecycle` (`RAXIS_LIVE_E2E=1 RAXIS_LIVE_E2E_REALISTIC=1`).
  Every step lists the file:line that runs, what state it expects, what it
  mutates, the audit event(s) it emits, and the failure modes that could
  surface (each annotated with [BUG-CANDIDATE] when worth a second look).

  Read order
  ----------
  Top-to-bottom is the wall-clock order of operations. The kernel daemon
  runs in parallel with the harness from §3 onwards; per-phase concurrency
  is called out inline.

  Conventions
  -----------
  HARNESS-N   step on the test process (cargo test)
  KERNEL-N    step on the spawned raxis-kernel daemon
  VM-N        step on a microVM guest (orchestrator/executor/reviewer)
  WIT-N       witness assertion at end-of-run
  [BUG-N]     numbered bug candidate for later triage

  S-2 : Audit chain seq resume + prev_sha256
       ────────────────────────────────────
       At Step 7a of boot, last_chain_state scans segment-000.jsonl line
       by line:
       * compute sha256 over each JSON line
       * track prev_sha256 = sha256(previous line)
       * return (next_seq, prev_sha256) of the last record
       If the file is missing/empty → (0, None) genesis
       If a malformed line is found → Err → exit AuditChainBroken
       [HAZARD] The scan is line-based and does NOT validate that
       prev_sha256 in each record matches sha256(previous line). That
       validation is in verify_chain_full, called by recovery::reconcile
       BEFORE last_chain_state. So the resume value is trusted to match
       the chain's tail; a corrupted segment after recovery passes would
       resume on a broken tail. Belt-and-braces: verify_chain_full re-runs
       on every reboot.

  S-3 : Notifications dispatch — kernel-owned writes even when silenced
       ────────────────────────────────────────────────────────────────
       In notifications/mod.rs::dispatch, the inbox.jsonl append AND the
       SQLite notifications row write happen BEFORE the route is consulted.
       Only after both writes does the fan-out check `Some([])` → return.
       This is intentional (kernel ground-truth) but: a silenced route
       still produces inbox + SQLite rows. Operators reading inbox.jsonl
       see "silenced" notifications. Behavioural quirk to document.

  S-4 : LogicalDeadlock FK satisfaction Tier 1 → Tier 2 → None
       ───────────────────────────────────────────────────────
       In orch_respawn_ceiling::insert_logical_deadlock_escalation_in_tx:
       * Tier 1: most-recently FSM-touched task whose session_id is NOT NULL
       * Tier 2: any task on initiative + most-recent Orchestrator session
       * fallthrough: return Ok(None) → escalation row NOT inserted
       This means INV-ESCALATION-AUTO-LOGICAL-DEADLOCK-PAIRED-WRITE-01 can
       silently degrade if BOTH tiers fail (no tasks at all). The kernel
       still marks the initiative Failed and emits the audit row, but
       there's no operator-actionable escalation. Defensive only — in
       practice every initiative has at least one task row.

  S-5 : Shutdown race on in-flight operator IPC frames
       ────────────────────────────────────────────────
       wait_for_shutdown returns on SIGTERM; abort_handle().abort() then
       cancels each accept loop. An operator connection that's mid-frame
       (dispatch_loop reading from socket) gets its task aborted, the
       socket file removed, and the in-flight reply is dropped without
       a graceful close. Operator CLI sees a connection-reset on the
       wire. Cleaner shutdown would: stop accepting new connections,
       drain existing ones with a deadline. V2 doesn't promise this and
       operators handle reconnect. Live-e2e never racey because the
       harness drops the OperatorIpc before SIGTERM.

  S-6 : Drop ordering: Tier3Reporter / KernelInstance / OtelPusherSupervisor
       ───────────────────────────────────────────────────────────────────
       Drop runs in REVERSE declaration order in Rust. In the test
       function locals:
         tier3                 declared LAST  → drops FIRST
         kernel                                → drops SECOND
         _otel_pusher_supervisor → drops THIRD
       Under success: tier3.mark_success() before scope end → Drop runs
       the success branch (no extra noise).
       Under panic: tier3.Drop emits the artifact block; kernel.Drop
       SIGKILLs the kernel if not already exited; otel_supervisor.Drop
       SIGTERMs + SIGKILLs the pusher.
       Under keep-alive: all three drops short-circuit to no-op.
       [HAZARD] If the panic happens AFTER tier3.mark_success() but
       BEFORE the natural scope-end (e.g. inside an assert in WIT-7.x),
       tier3.Drop still emits the artifact block because mark_success
       only affects the workdir-keep policy, not the block emission. OK.

  S-7 : poll loop stderr scan re-reads entire kernel.stderr.log every 500ms
       ───────────────────────────────────────────────────────────────────
       At kernel_driver.rs::scan_stderr_for_terminal_spawn_failure each
       iteration: std::fs::read(path) → split('\n') → for-each memmem.

       tens of ms per iteration. Mitigation: keep a byte-offset cursor
       between iterations and only scan new bytes. Worth fixing — at
       least bound the read with a tail-only mode.

  S-8 : enable_gateway_in_policy assumes EXACTLY ONE [gateway] block
       ────────────────────────────────────────────────────────────
       The helper asserts !body.contains("\n[gateway]\n") before appending.
       If a future policy template ships with [gateway] (e.g. for an
       in-tree dev gateway), the helper panics with "bootstrap template
       changed". Documented; safe.

  S-9 : Operator IPC handshake captures policy snapshot for entire flow
       ────────────────────────────────────────────────────────────────
       After authentication, dispatch_loop runs against an Arc<HandlerContext>
       whose `policy` is the Arc<ArcSwap<PolicyBundle>>. Each handler
       (handle_create_initiative, handle_approve_plan) calls policy.load()
       or policy.load_full() to pin its OWN snapshot. So policy hot-reloads
       between two CreateInitiative+ApprovePlan pairs on the SAME connection
       can result in the two calls running against different epochs. The
       SUBMIT step might validate against epoch N; APPROVE against epoch
       N+1. Each invocation is internally consistent (snapshot pinned)
       but cross-invocation drift is possible. Documented by the bundle
       reload model; not a defect in V2.

  S-10: Per-channel notification SQLite insert uses spawn_blocking +
        lock_sync, but the equivalent test helper uses lock().await
        ───────────────────────────────────────────────────────────
        Production: tokio::task::spawn_blocking + Store::lock_sync
        Test:       Store::lock().await
        Both paths are correct under the lock-sync invariant. The test
        helper migration is the canonical example of the new
        lock().await preferred API.
