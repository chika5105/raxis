# raxis perf report — 2026-05-12

This is the multi-section perf report that consolidates every
`cargo xtask perf` run for the day. Per
`specs/v3/observability-prometheus.md §10`, the canonical filename
is `perf-report-YYYY-MM-DD.md` and is regenerated by every `cargo
xtask perf all` invocation; ad-hoc per-subcommand runs ALSO emit a
focused report alongside (`vm-cold-boot-YYYY-MM-DD.md`,
`audit-throughput-YYYY-MM-DD.md`, ...) so a follow-up bisect can
see the per-subcommand numbers without parsing this file.

## Environment

- **host**          macOS Darwin 25.5.0 (Apple Silicon)
- **rustc**         workspace pin (`raxis/rust-toolchain`)
- **build profile** `dev` (debug — release builds disable the
  `SubprocessIsolation` substrate behind `cfg(any(debug_assertions,

- **observability stack** live-e2e

  Prometheus `v2.55.1`, Grafana `11.3.0`, OTel collector
  `contrib:0.110.0`. 14-day retention. Named volumes
  `raxis-live-e2e-test_prometheus_data` /
  `raxis-live-e2e-test_grafana_data`.

## Section 1 — VM cold-boot (substrate=subprocess, n=500)

```
metric (ms)                   p50      p95      p99
cold_boot                    0.11     0.17     0.28
host_init                    0.00     0.00     0.00
guest_init                   0.11     0.17     0.28
wall-clock total: 0.08s (6634.2 spawns/s)
```

The `subprocess` substrate execs `/bin/cat` and returns as soon as
`Command::spawn` resolves; the ~110 µs floor reflects the host's
`posix_spawn` syscall plus tokio's `spawn_blocking` task hop.
**This is NOT a microVM cold-boot number** — it is the test
substrate floor that the four-tier histogram family is calibrated
against. Once the AVF prereqs (`cargo xtask dev-prereqs`,
`cargo xtask images dev-stage`) are staged, `cargo xtask perf
vm-cold-boot --backend apple-vz` will report Apple-VZ numbers
against the same histogram family — and the
`raxis-10-isolation` Grafana dashboard will pivot the same charts
by the `backend` label so the operator can compare both substrates
side-by-side.

The four-tier histogram emits per spawn through
`SessionSpawnService` and lands at the OTel collector via
`raxis-otel-pusher` -> Prometheus on the live-e2e stack:

- `raxis.isolation.spawn.cold_boot.duration` — user-visible total
- `raxis.isolation.spawn.host_init.duration` — pre-`Backend::spawn`
- `raxis.isolation.spawn.guest_init.duration` — substrate-bounded
- `raxis.isolation.spawn.vsock_handshake.duration` — substrate-internal
  (left unset by `SessionSpawnService`; AVF / Firecracker
  substrates self-report it in a follow-up patch)

## Section 2 — Audit append throughput (n=1000)

```
metric (us)                   p50      p95      p99
append latency               4067     6132     7136
throughput =     229.7 appends/s (wall-clock: 4.35s)
```

Each append is one JSONL line plus a `fsync_data()` (mirroring
`audit_sync_on_append: true` in production). The macOS APFS
`fsync` floor is the dominant cost; that 4-7 ms per append
matches what the audit-chain integration tests see.

The matching Prometheus series is
`raxis_audit_event_append_duration_milliseconds_*` (Grafana
dashboard `raxis-30-audit`).

## How to reproduce

```bash
# 1. Bring up the live-e2e Prometheus + Grafana + OTel stack:
docker compose -f raxis/live-e2e/docker-compose.e2e.yml up -d --wait \
    otel-collector prometheus grafana

# 2. Run every benchmark:
cd raxis && cargo xtask perf all --iterations 500

# 3. Open Grafana to inspect the time-series (anonymous Viewer):
open 'http://127.0.0.1:3000/d/raxis-00-overview'

# 4. Wipe the persisted data when you are done (volumes survive
#    `docker compose down` by default):
docker compose -f raxis/live-e2e/docker-compose.e2e.yml down
docker volume rm raxis-live-e2e-test_prometheus_data raxis-live-e2e-test_grafana_data
```
