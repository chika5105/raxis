# Why Not Just Cursor Agents + Spec File?

The short answer: a spec file is an instruction document. RAXIS is an enforcement layer. Instructions require trust; enforcement doesn't.

---

## The core problem: intelligence and authority cannot safely share a process

When you give a Cursor agent a spec file, the agent reads the spec, forms an interpretation, and acts on it. At every step the agent is both the worker and the judge of whether it's complying. That's circular. A sufficiently confident LLM that has partially misunderstood a spec will comply with its own interpretation — which may or may not match yours. You have no way to distinguish "agent followed the spec" from "agent convinced itself it followed the spec."

RAXIS is designed around the principle that the entity doing the work (the planner) and the entity enforcing constraints (the kernel) cannot be the same process. The kernel is not an LLM. It doesn't have opinions about the spec. It evaluates typed predicates against deterministic inputs.

---

## Problem by problem

### 1 — Verification theater

If you spin up Agent A to write code and Agent B to review it, you've created a shared-delusion loop. Both are LLMs with the same failure modes: hallucination, sycophancy, context length limits, confident wrongness. A well-phrased rationalization from Agent A will often satisfy Agent B — not because the code is correct, but because the argument is coherent. This is A.5 in the README's rejected alternatives section.

RAXIS replaces LLM reviewers with deterministic witnesses: exit codes, test pass records, static analysis results — all cryptographically bound to a specific commit SHA and task ID. The witness is generated by a subprocess that runs a real tool, not by an LLM evaluating the output of another LLM. The kernel checks whether a valid witness record exists for the right commit before the task completes. This is structurally independent of the planner's reasoning.

### 2 — Authority is whatever the agent decides it is

When a Cursor agent modifies `src/auth/session.rs`, who authorized that? The spec did — but the spec is text the agent interpreted. The agent decided its interpretation permits touching that file. It might be right; it might be rationalizing.

RAXIS derives touched paths from VCS diff (not from anything the agent supplies), evaluates claim requirements from the signed policy artifact, and checks whether the agent's session holds valid claims for those paths. The agent never interprets its own authority. If the policy says `src/auth/**` requires `SecurityReview` claim, the kernel enforces that regardless of what the agent believes the spec says.

### 3 — The context window is not a trust boundary

A spec file lives in the agent's context. The agent can selectively attend to it, misweigh it against other context, or simply lose track of it as the conversation grows. RAXIS keeps all enforcement state outside the agent's context in the kernel state store — the agent's context is completely irrelevant to admission, gate evaluation, and budget enforcement decisions.

### 4 — Budget runaway

Multiple Cursor agents running against a complex task have no shared budget control. Each one issues inference calls, web fetches, and tool invocations independently. The natural stopping condition is "all agents finish" — which might happen after enormous spend, or not at all. You'd need something external (a billing alert, a human watching) to intervene.

RAXIS pre-computes admission cost at intent time, reserves budget against the lane ceiling, and refuses to admit a task that would exceed the ceiling. Budget enforcement is structural, not behavioral — it doesn't rely on agents being cost-conscious.

### 5 — Unauthorized egress

A Cursor agent with web access can call arbitrary URLs. In a multi-agent setup this is significant: an agent could be prompted to make API calls to external services, exfiltrate code in a URL parameter, or stage data outside your workspace. The spec says "don't do this" — but there's no mechanism that enforces it structurally.

RAXIS's planner process has no HTTP client in its dependency closure (enforced at compile time by `Cargo.toml` audit). Every external fetch goes through the kernel → gateway pipeline, with the kernel enforcing the domain allowlist and rate limit before the gateway makes any network call. An agent literally cannot make an unauthorized fetch — the capability doesn't exist in the binary.

### 6 — Correlated failures

Multiple LLM agents share failure modes. They'll make the same kinds of mistakes on the same kinds of inputs — not independently, but correlated. Running 10 agents doesn't give you 10x the safety checks; it gives you 10 entities that might all agree on the same wrong answer. RAXIS's verifier is structurally different from the planner. Lint failures and test failures are not subject to persuasion. A test that fails cannot be talked into passing.

### 7 — No recoverable audit trail

If a multi-agent Cursor run produces a bad output, what happened? You have conversation history (if you saved it) and file diffs. Reconstructing what decisions were made and why is detective work. RAXIS's append-only, chain-verified audit log records every decision: session creation, intent submission, gate evaluation result, witness accepted, budget consumed, escalation submitted, policy epoch advanced. You can replay an entire initiative from the audit log and verify that the chain of decisions was consistent.

### 8 — Scope creep under pressure

LLM agents expand scope when they hit obstacles. "I can't write this test without also refactoring that other module" cascades into a much larger changeset than the original intent. The spec doesn't prevent this because the agent decides what the spec requires. RAXIS's VCS-derived path evaluation means the kernel knows exactly what files were touched, compares that against the pre-declared scope, and fires the claim gate if the diff exceeds what the session's claims cover. The agent cannot rationalize its way to additional scope.

### 9 — Escalation is unstructured

When a Cursor agent hits something it can't or shouldn't do — needs a secret, needs to modify a restricted file, needs operator input — it either tries anyway, stops silently, or posts a message. None of these are auditable, recoverable, or scoped. RAXIS's escalation FSM is a first-class control plane event: the agent submits a typed escalation, the kernel records it, the operator is notified, the operator issues a cryptographically scoped approval token with an explicit TTL and predicate, the kernel validates the token before allowing the action. If the operator never responds, the escalation times out — the system stops at a defined boundary, not at whatever the agent decides to do next.

## The asymmetric cost argument

"Just use Cursor agents" has near-zero design cost: write a spec, press go. RAXIS has real upfront cost: design the policy artifact, define gate requirements, write the claim table, run the signing ceremony.

The tradeoff flips at scale and complexity. A single well-defined task with one agent and a short context window is probably fine with just a spec. As you add agents, increase task complexity, extend runtime, or touch security-sensitive code, the "just trust the agent" approach accumulates implicit risk that's invisible until something goes wrong. RAXIS pays the design cost once and gets runtime safety that doesn't degrade as you add agents or extend the initiative duration.

The RAXIS design principles section (A.1) puts it directly: *"intelligent and authoritative are incompatible properties in the same process when the inputs are untrusted."* The spec file approach merges them. RAXIS separates them by construction.

## Grounding in proven security models: privilege separation

RAXIS's process architecture is not a novel idea — it is a deliberate application of **privilege separation**, a security principle with a long track record in systems software.

The canonical examples:

- **OpenSSH** splits into a privileged monitor process and an unprivileged child process that handles the untrusted network connection. The monitor holds the private keys and makes authentication decisions; the child does the parsing and I/O. A compromise of the child cannot yield the private keys because they are never in its address space.

- **Google Chrome** runs each tab in a sandboxed renderer process with no direct OS access. The browser kernel process mediates all privileged operations (file I/O, network, GPU). A compromised renderer cannot escalate to OS-level capabilities without defeating the kernel layer.

- **OpenBSD's `sshd` and `httpd`** apply the same model with `pledge(2)` and `unveil(2)`: processes declare the minimal set of capabilities they need at startup, and the kernel enforces that ceiling for the rest of the process lifetime. Any capability not declared is unavailable, regardless of what the code tries to do.

The common structure in all of these: **an unprivileged worker process handles untrusted input; a privileged supervisor process makes authority decisions; the two communicate through a narrow, typed channel the supervisor controls.**

RAXIS applies this exactly:

| Security model | Unprivileged worker | Privileged supervisor | Channel |
|---|---|---|---|
| OpenSSH | Unprivileged child (network I/O) | Monitor (key operations) | Pipe with typed messages |
| Chrome | Renderer (untrusted content) | Browser kernel (OS access) | Mojo IPC |
| RAXIS | Planner (LLM inference, proposals) | Kernel (authority, budgets, gates) | UDS with typed IPC envelope |

The novel aspect in RAXIS is that the "untrusted input" is not a network packet or a web page — it is the output of a language model. LLM output is untrusted for the same structural reason that network input is untrusted: it comes from outside the trust boundary and can be manipulated, hallucinated, or adversarially crafted. Treating it with the same discipline that system security engineers apply to network input is not paranoia — it is applying a model that has proven correct in practice for decades.

The spec file approach has no equivalent of the monitor process or the browser kernel. The agent is simultaneously the renderer and the browser kernel, the child and the monitor. Every security principle that justifies privilege separation applies equally here — and the absence of that separation is the structural gap RAXIS fills.
