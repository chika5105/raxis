# Scenario 50 — End-to-End Feature Shipment

> **Complexity:** ⭐⭐⭐⭐ Expert | **Wall clock:** ~30 min | **Provider:** Anthropic

The capstone scenario. A realistic feature shipment uses every major
feature: a parallel decomposition, a panel review, a sequential
refinement, mechanical witnesses (build, test, clippy, curl-against-
dev-server), `path_allowlist`, `allowed_egress`, and an explicit
operator approval gate.

This is the *demo-day* scenario: showcase the system end to end.

---

## Prerequisites

- Same as scenarios 04 and 15 combined.
- A working PostgreSQL is *not* required for this scenario; the demo
  uses an in-memory SQLite fixture instead.

---

## What this scenario demonstrates

1. **Genesis check** — confirm the kernel is alive.
2. **Plan validate** — structural pre-flight.
3. **Plan submit + approve** — operator-gated lifecycle.
4. **Parallel decomposition** — backend Executor + frontend Executor
   in parallel.
5. **Sequential refinement** — Implementer → Tester → Reviewer.
6. **Panel review** — three Reviewers debate quality.
7. **Mechanical witnesses** — `cargo build`, `cargo test`, `curl`
   against an in-VM dev server.
8. **`path_allowlist`** — backend can't touch frontend; frontend
   can't touch backend.
9. **`allowed_egress`** — only `api.github.com:443`.
10. **IntegrationMerge** — the Orchestrator merges all branches
    deterministically.

## Repository setup

```bash
export DEMO_ROOT="/tmp/raxis-scenario-50"
rm -rf "$DEMO_ROOT" && mkdir -p "$DEMO_ROOT"
cd "$DEMO_ROOT"

cargo new --bin --name demo50 -q
cargo add tiny_http -q
mkdir -p frontend/src docs tests
cat > frontend/src/index.js <<'JS'
// auto-generated by demo50
JS

git -c user.email=demo@raxis.local -c user.name=Demo init -q
git -c user.email=demo@raxis.local -c user.name=Demo add . > /dev/null
git -c user.email=demo@raxis.local -c user.name=Demo commit -qm "init"
```

---

## Genesis check

```bash
raxis kernel status || (echo "Kernel not running"; exit 1)
raxis genesis status || raxis genesis run --no-dry-run
```

---

## Run it

```bash
raxis plan validate ./plan.toml
raxis submit plan ./plan.toml --no-dry-run

# Capture INIT_ID:
INIT_ID="$(raxis initiative list --state Draft --json | jq -r '.[0].initiative_id')"
echo "INIT_ID=$INIT_ID"

# Approve the plan:
raxis plan approve "$INIT_ID"

# Stream live progress:
watch -n 2 "raxis initiative show $INIT_ID --with-tasks"
```

---

## What "success" looks like

- `raxis initiative show "$INIT_ID"` ends with `state = Completed`.
- `git log --oneline` in the workspace shows commits from each agent.
- `raxis audit verify --initiative-id "$INIT_ID"` returns
  `chain ok`.

---

## Tear-down

```bash
raxis initiative abort "$INIT_ID" 2>/dev/null || true
rm -rf "$DEMO_ROOT"
```
